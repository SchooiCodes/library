What is Redis?

Redis (Remote Dictionary Server) is an open-source, in-memory data structure store used as a database, cache, message broker, and streaming engine. It is known for its sub-millisecond response times due to keeping the entire dataset in RAM.

Redis supports a variety of data structures including strings, hashes, lists, sets, sorted sets, bitmaps, hyperloglogs, geospatial indexes, and streams. It is written in ANSI C and is designed for high performance and simplicity.

Redis is used by millions including Twitter, GitHub, Snapchat, and Stack Overflow for caching, real-time analytics, leaderboards, session stores, and message queues.

Installation

Install Redis using your package manager or Docker:

# Debian / Ubuntu
sudo apt update
sudo apt install redis-server

# RHEL / Fedora
sudo dnf install redis

# macOS (Homebrew)
brew install redis

# Docker
docker run --name redis -d -p 6379:6379 redis:7-alpine

# Start the service
sudo systemctl enable redis-server --now

# Verify it's running
redis-cli ping
# Should return: PONG

Redis runs on port 6379 by default. Configuration file is at /etc/redis/redis.conf (or /etc/redis.conf depending on distribution).

Tip: Use redis-cli --stat for a live stats dashboard of your Redis server.

Basic Commands

Redis commands are simple, text-based protocol commands. Here are the most essential ones:

# String operations
SET name "Alice"
GET name
DEL name
EXISTS name
STRLEN name

# Keyspace management
KEYS *     # List all keys (avoid in production - use SCAN)
EXISTS mykey
TYPE mykey
RENAME mykey newkey
TTL mykey    # Time to live in seconds

# Increment/Decrement
SET counter 10
INCR counter  # 11
INCRBY counter 5 # 16
DECR counter  # 15

# Checking/Setting multiple keys
MSET a 1 b 2 c 3
MGET a b c

Redis commands are atomic and run sequentially on a single-threaded event loop. There are over 200 commands organized by data type.

Data Types

Redis supports multiple data structures beyond simple key-value pairs:

# Strings (most basic type)
SET welcome "Hello, World!"

# Lists (ordered, allow duplicates)
LPUSH tasks "task1"
RPUSH tasks "task2"
LPOP tasks
LRANGE tasks 0 -1

# Sets (unordered, no duplicates)
SADD tags "redis" "database" "cache"
SMEMBERS tags
SISMEMBER tags "redis"

# Hashes (field-value pairs)
HSET user:1000 name "Alice" age "30" city "NYC"
HGET user:1000 name
HGETALL user:1000

# Sorted Sets (unique elements with scores)
ZADD leaderboard 100 "player1" 85 "player2" 200 "player3"
ZRANGE leaderboard 0 -1 WITHSCORES
ZREVRANGE leaderboard 0 2  # Top 3

# Bitmaps
SETBIT user:signin:2024-01 123 1
GETBIT user:signin:2024-01 123
BITCOUNT user:signin:2024-01

Expiration & TTL

A key Redis feature is automatic key expiration. You can set a time-to-live (TTL) on any key:

# Set a key that expires in 60 seconds
SET session:abc123 "user_data"
EXPIRE session:abc123 60

# Set with expiration inline
SETEX session:abc123 60 "user_data"

# Check remaining TTL
TTL session:abc123

# Remove expiration
PERSIST session:abc123

# Set expiration at a specific timestamp
EXPIREAT session:abc123 1700000000

# Millisecond precision
PSETEX session:abc123 60000 "user_data"
PTTL session:abc123

Expired keys are removed lazily (when accessed) and periodically (every 100ms). This makes Redis ideal for caching and temporary session data.

Persistence (RDB/AOF)

Redis offers two persistence mechanisms to save in-memory data to disk:

# RDB (Redis Database) - Point-in-time snapshots
# Configure in redis.conf:
save 900 1    # Snapshot if at least 1 key changed in 900s
save 300 10   # Snapshot if at least 10 keys changed in 300s
save 60 10000  # Snapshot if at least 10000 keys changed in 60s
dbfilename dump.rdb
dir /var/lib/redis

# AOF (Append Only File) - Logs every write operation
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec  # Balance between speed and durability
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

You can use RDB-only for caching (some data loss is acceptable), AOF-only for durability, or both for the best of both worlds. Redis Enterprise also supports replication with automatic failover.

Pub/Sub

Redis Pub/Sub (Publish/Subscribe) is a lightweight messaging pattern for real-time communication:

# Terminal 1 - Subscriber
SUBSCRIBE notifications

# Terminal 2 - Publisher
PUBLISH notifications "New user registered!"
PUBLISH notifications "Order #1234 shipped"

# Pattern matching subscriptions
PSUBSCRIBE news:*   # Subscribes to news:sports, news:tech, etc.

# Message queues with lists (more reliable than Pub/Sub)
LPUSH job_queue "process_image"
BRPOP job_queue 0   # Blocking pop, waits indefinitely

Pub/Sub delivers messages to all active subscribers but does not persist messages for disconnected clients. For reliable queuing with persistence, use Redis Streams or a list-based queue pattern with BRPOPLPUSH.

Redis CLI

The redis-cli tool is powerful for administration and debugging:

# Interactive mode
redis-cli

# Run single command
redis-cli GET mykey

# Monitor all commands in real-time
redis-cli MONITOR

# Scan keys safely (production-safe alternative to KEYS *)
redis-cli --scan --pattern "session:*"

# Check server info
redis-cli INFO

# Benchmark Redis performance
redis-benchmark -n 100000 -q

# Slow log (queries slower than 10ms)
redis-cli SLOWLOG GET 10
redis-cli SLOWLOG RESET

# Connect to remote server
redis-cli -h redis.example.com -p 6379 -a password

Security & Authentication

Redis is designed for trusted environments. For production, always enable security features:

# redis.conf security settings
bind 127.0.0.1 ::1   # Only listen on localhost
port 6379
requirepass very_strong_password
rename-command FLUSHALL ""
rename-command FLUSHDB ""
rename-command CONFIG ""
rename-command SHUTDOWN "SHUTDOWN_MYSELF"

# ACL-based access control (Redis 6+)
aclfile /etc/redis/users.acl

# Sample users.acl
user default off     # Disable default user
user alice on >password ~* +@all -@dangerous
user billing on >secure_pass ~billing:* +get +set +hget +hset

Additional security measures: run Redis behind a firewall, never expose it directly to the internet, use TLS/SSL with tls-port 6380, and always set a strong password or use ACLs.

Use Cases (Caching, Queues, Sessions)

Redis excels in several common production scenarios:

  • Caching: Store expensive query results or API responses with TTL. Cache-aside pattern: check Redis first, fetch from DB on miss, store in Redis for next time.
  • Session storage: Store user sessions with automatic expiration using EXPIRE. Much faster than database-backed sessions.
  • Rate limiting: Use INCR with EXPIRE to track API request counts per IP or user within a time window.
  • Message queues: Use lists with LPUSH / BRPOP for simple queue patterns, or Redis Streams for consumer groups.
  • Leaderboards: Sorted sets (ZADD, ZREVRANGE) are perfect for real-time ranking systems in gaming or analytics.
  • Real-time counters: Track page views, likes, or any event count with INCR.
  • Geospatial: Store locations with GEOADD and query nearby points for food delivery, ride-sharing, or store locators.
# Caching example
GET product:123      # Check cache
# ... cache miss ...
SETEX product:123 3600 "{\"name\":\"Widget\",\"price\":9.99}"

# Rate limiting example
INCR ratelimit:192.168.1.1:api
EXPIRE ratelimit:192.168.1.1:api 60
