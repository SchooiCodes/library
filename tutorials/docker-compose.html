💠 Docker Compose

Define and run multi-container Docker applications with ease using Docker Compose.

What is Docker Compose?

Docker Compose is a tool for defining and running multi-container Docker applications. With a single docker-compose.yml file, you define your application's services, networks, and volumes, then spin everything up with one command.

Compose solves the problem of managing multiple containers that need to communicate. Instead of running half a dozen docker run commands with complex flags, you declare the entire stack in a YAML file and use simple CLI commands like docker compose up and docker compose down.

Docker Compose is ideal for development environments, CI/CD pipelines, and single-host production deployments. It was originally a separate Python tool but is now integrated into Docker itself as a plugin.

docker-compose.yml Structure

The docker-compose.yml file uses YAML syntax and has three top-level keys: services, networks, and volumes. Here is the basic structure:

version: "3.9"

services:
 web:
  image: nginx:latest
  ports:
   - "80:80"

 app:
  build: .
  ports:
   - "3000:3000"
  depends_on:
   - db

 db:
  image: postgres:15
  volumes:
   - pgdata:/var/lib/postgresql/data

volumes:
 pgdata:

networks:
 default:

Indentation matters in YAML — use 2 spaces per level. The version key is optional in newer Docker versions but recommended for clarity.

Tip: Always validate your YAML with docker compose config before deploying. This command parses your compose file and shows the resolved configuration.

Services

Services are the core of Docker Compose — each service corresponds to a container. You define services under the services: key and configure them with image, build, ports, volumes, environment variables, and more.

services:
 api:
  build:
   context: .
   dockerfile: Dockerfile.dev
  ports:
   - "8080:8080"
  environment:
   - NODE_ENV=development
  volumes:
   - .:/app
   - /app/node_modules
  command: npm run dev

Key service options include image (pull from registry), build (build from Dockerfile), ports (port mapping), volumes (persistent storage), environment (env variables), depends_on (startup order), and command (override CMD).

Networks

Compose automatically creates a default network for your application, so all services can reach each other by service name. You can also define custom networks for isolation:

services:
 frontend:
  image: my-frontend
  networks:
   - frontend
   - backend

 api:
  image: my-api
  networks:
   - backend

 db:
  image: postgres
  networks:
   - backend

networks:
 frontend:
 backend:

Custom networks let you control which services can communicate. For example, the frontend can reach both frontend and backend networks, while the database only attaches to backend for security.

You can also configure network drivers (bridge, host, overlay for Swarm) and set static IPs with ipam configurations.

Volumes

Volumes provide persistent storage for containers. Without volumes, data is lost when a container restarts. Compose makes it easy to define and mount volumes:

services:
 mysql:
  image: mysql:8
  volumes:
   - mysql_data:/var/lib/mysql
   - ./init.sql:/docker-entrypoint-initdb.d/init.sql:ro
  environment:
   MYSQL_ROOT_PASSWORD: secret

volumes:
 mysql_data:

There are three types of mounts in Compose: named volumes (managed by Docker), bind mounts (host path), and tmpfs (in-memory, for temporary data). Named volumes persist even after docker compose down, while bind mounts share files between host and container.

Use named volumes for databases and other stateful services. Use bind mounts for development code syncing.

Environment Variables

Environment variables keep your configuration portable and secure. Compose supports multiple ways to set them:

services:
 app:
  image: my-app
  # Inline environment variables
  environment:
   - DEBUG=true
   - LOG_LEVEL=info
  
  # Or use env_file
  env_file:
   - .env
   - .env.production
  
  # Variable substitution in compose file
  image: my-app:${TAG:-latest}

Use a .env file in the same directory as your compose file to store variables. Compose automatically reads it. For sensitive values like passwords, consider using Docker secrets or an external vault.

Variable substitution uses shell-like syntax: ${VAR} for required, ${VAR:-default} with fallback, and ${VAR:?error} to fail if unset.

Health Checks

Health checks allow Docker to monitor whether a service is actually ready, not just the container process running. Compose supports the same health check syntax as Dockerfiles:

services:
 db:
  image: postgres:15
  healthcheck:
   test: ["CMD-SHELL", "pg_isready -U postgres"]
   interval: 10s
   timeout: 5s
   retries: 5
   start_period: 30s

 app:
  build: .
  depends_on:
   db:
    condition: service_healthy

The depends_on condition: service_healthy ensures your app waits until the database passes its health check before starting.

Common health checks: curl -f http://localhost for web services, pg_isready for PostgreSQL, redis-cli ping for Redis, and mysqladmin ping for MySQL.

Common Compose Files

Here is a complete example of a full-stack web app with React, Node.js, PostgreSQL, and Redis:

services:
 frontend:
  build: ./client
  ports:
   - "3000:3000"
  depends_on:
   - api
  environment:
   - REACT_APP_API_URL=http://api:5000

 api:
  build: ./server
  ports:
   - "5000:5000"
  depends_on:
   db:
    condition: service_healthy
   redis:
    condition: service_started
  environment:
   - DATABASE_URL=postgres://user:pass@db:5432/mydb
   - REDIS_URL=redis://redis:6379

 db:
  image: postgres:15
  volumes:
   - pgdata:/var/lib/postgresql/data
  environment:
   POSTGRES_USER: user
   POSTGRES_PASSWORD: pass
   POSTGRES_DB: mydb

 redis:
  image: redis:7-alpine

volumes:
 pgdata:

This pattern — separate services for frontend, API, database, and cache — is extremely common in modern web development. Each service can be developed independently and scaled separately.

CLI Commands

The docker compose CLI provides commands to manage your application lifecycle:

CommandDescription
docker compose up -dStart all services in detached mode
docker compose downStop and remove containers, networks
docker compose logs -fFollow log output from all services
docker compose psList running containers
docker compose exec <service> bashOpen a shell in a running container
docker compose buildBuild or rebuild services
docker compose pullPull latest images
docker compose restartRestart all services
docker compose configValidate and view resolved config
docker compose topShow running processes per service

Production Considerations

While Compose is great for development, production deployments require additional considerations:

  • Use Docker Swarm or Kubernetes for multi-host deployments. Compose is designed for single hosts.
  • Set resource limits with deploy.resources.limits to prevent containers from consuming all host memory.
  • Use restart policies like restart: unless-stopped to ensure containers recover from crashes.
  • Don't hardcode secrets — use Docker secrets, environment variables from a vault, or .env files excluded from version control.
  • Pin image versions instead of using latest tags to ensure reproducible deployments.
  • Set up logging with logging: driver configuration to ship logs to a central system like ELK or Grafana Loki.
  • Use health checks for zero-downtime deployments and orchestration-aware container management.
