🔄 CI/CD Pipeline Guide

Build, test, and deploy automatically with continuous integration and continuous delivery pipelines.

What is CI/CD?

CI/CD (Continuous Integration/Continuous Delivery) automates the process of building, testing, and deploying your code. Every time you push to a repository, a pipeline runs your tests, builds artifacts, and deploys to staging or production — all without manual intervention.

Continuous Integration (CI): Automatically build and test every commit. Catch bugs early, enforce code quality, and ensure the main branch is always deployable.

Continuous Delivery (CD): Automatically deploy tested code to staging, with manual approval for production. Continuous Deployment goes further — every passing commit goes straight to production.

Core Concepts

  • Pipeline — A series of stages (build → test → deploy) that run sequentially or in parallel
  • Stage — A logical phase (e.g., "Build", "Unit Tests", "Integration Tests", "Deploy")
  • Job/Step — Individual commands within a stage (e.g., npm install, npm test)
  • Trigger — What starts the pipeline (push, PR, schedule, manual)
  • Artifact — Files produced by a stage (binaries, packages, reports) passed to later stages
  • Environment — Where the pipeline runs (GitHub Actions runner, self-hosted server, Kubernetes pod)

GitHub Actions Pipeline

# .github/workflows/ci.yml
name: CI/CD Pipeline

on:
 push:
  branches: [main, develop]
 pull_request:
  branches: [main]

jobs:
 test:
  runs-on: ubuntu-latest
  steps:
   - uses: actions/checkout@v4
   - uses: actions/setup-node@v4
    with:
     node-version: 20
     cache: 'npm'
   - run: npm ci
   - run: npm run lint
   - run: npm run typecheck
   - run: npm test -- --coverage
   - uses: actions/upload-artifact@v4
    with:
     name: coverage
     path: coverage/

 build:
  needs: test
  runs-on: ubuntu-latest
  steps:
   - uses: actions/checkout@v4
   - run: npm ci
   - run: npm run build
   - uses: actions/upload-artifact@v4
    with:
     name: build
     path: dist/

 deploy:
  needs: build
  if: github.ref == 'refs/heads/main'
  runs-on: ubuntu-latest
  steps:
   - uses: actions/download-artifact@v4
    with:
     name: build
     path: dist/
   - run: |
     rsync -avz --delete dist/ user@server:/var/www/app/
    env:
     SSH_PRIVATE_KEY: ${{ secrets.DEPLOY_KEY }}

GitLab CI Pipeline

# .gitlab-ci.yml
stages:
 - test
 - build
 - deploy

variables:
 DOCKER_IMAGE: $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA

cache:
 key: ${CI_COMMIT_REF_SLUG}
 paths:
  - node_modules/

before_script:
 - npm ci

test:
 stage: test
 script:
  - npm run lint
  - npm run typecheck
  - npm test -- --coverage
 artifacts:
  reports:
   coverage_report:
    coverage_format: cobertura
    path: coverage/cobertura-coverage.xml

build:
 stage: build
 script:
  - docker build -t $DOCKER_IMAGE .
  - docker push $DOCKER_IMAGE
 only:
  - main

deploy-prod:
 stage: deploy
 script:
  - kubectl set image deployment/app app=$DOCKER_IMAGE
 environment:
  name: production
  url: https://app.example.com
 only:
  - main
 when: manual # Requires manual approval

Self-Hosted Runners

Self-hosted runners give you more control, cheaper builds (no per-minute billing), and access to internal network resources.

# GitHub Actions self-hosted runner (Linux)
mkdir actions-runner && cd actions-runner
curl -o actions-runner-linux-x64-2.320.0.tar.gz -L \
 https://github.com/actions/runner/releases/download/v2.320.0/actions-runner-linux-x64-2.320.0.tar.gz
tar xzf actions-runner-linux-x64-*.tar.gz

# Configure (get token from GitHub repo Settings → Actions → Runners)
./config.sh --url https://github.com/username/repo --token YOUR_TOKEN

# Run as a service
sudo ./svc.sh install
sudo ./svc.sh start

# GitLab runner
sudo docker run -d --name gitlab-runner --restart always \
 -v /var/run/docker.sock:/var/run/docker.sock \
 -v /etc/gitlab-runner:/etc/gitlab-runner \
 gitlab/gitlab-runner:latest

sudo docker exec -it gitlab-runner gitlab-runner register

CI/CD Best Practices

  • Fail fast — Run the fastest and most impactful checks (lint, typecheck) first. If they fail, stop the pipeline and save resources.
  • Cache dependencies — Speed up builds by caching node_modules, pip packages, Maven artifacts, etc.
  • Use secrets properly — Never hardcode credentials. Use CI/CD secret management for API keys, SSH keys, and passwords.
  • Pin versions — Pin runner versions, action versions, and tool versions to avoid unexpected breakage from upstream changes.
  • Parallelize stages — Run independent jobs in parallel. Matrix builds let you test across multiple OS/version combinations.
  • Keep pipelines fast — Aim for under 10 minutes. Split large tasks, use caching, and optimize Docker layers.
  • Test deployments — Deploy to staging first. Use blue-green or canary deployments for production.
  • Monitor and alert — Set up notifications for pipeline failures. Track deployment frequency and failure rate as metrics.
