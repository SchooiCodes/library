File Sharing with Samba & NFS

This guide covers setting up Samba (SMB/CIFS) and NFS file sharing on a Linux server, configuring shares, securing access, and connecting from Windows, macOS, and Linux clients.

1. Samba Installation & Configuration

Install Samba

# Debian / Ubuntu
sudo apt update
sudo apt install samba samba-common smbclient

# RHEL / Fedora
sudo dnf install samba samba-client samba-common

# Arch Linux
sudo pacman -S samba smbclient

Basic smb.conf

The main configuration file is /etc/samba/smb.conf. Below is a minimal example with a secured share:

[global]
   workgroup = WORKGROUP
   server string = File Server
   netbios name = FILESERVER
   security = user
   map to guest = bad user
   smb encrypt = required
   min protocol = SMB3
   server signing = required

[Shared]
   path = /srv/samba/shared
   browseable = yes
   read only = no
   guest ok = no
   valid users = @smbusers
   create mask = 0660
   directory mask = 0770
Tip: Use testparm to validate your smb.conf before restarting. Run sudo testparm to catch syntax errors.

Create Samba Users

# Create a system user (skip if user already exists)
sudo useradd -M -s /sbin/nologin alice

# Add to smbusers group
sudo groupadd smbusers
sudo usermod -aG smbusers alice

# Set Samba password (separate from system password)
sudo smbpasswd -a alice

# Enable and start Samba services
sudo systemctl enable smb nmb
sudo systemctl start smb nmb

2. NFS Setup

Install NFS Server

# Debian / Ubuntu
sudo apt install nfs-kernel-server

# RHEL / Fedora
sudo dnf install nfs-utils

# Arch Linux
sudo pacman -S nfs-utils

Configure /etc/exports

The /etc/exports file defines directories exported via NFS:

/srv/nfs/shared 192.168.1.0/24(rw,sync,no_subtree_check,no_root_squash)
/srv/nfs/backups 10.0.0.0/24(ro,sync,no_subtree_check)
/srv/nfs/media 192.168.1.100(rw,sync,no_subtree_check)
OptionDescription
rw/roRead-write or read-only access
syncWrite changes to disk before replying
no_subtree_checkDisable subtree checking (improves reliability)
no_root_squashAllow root on client to have root privileges (use with caution)
 
# Apply exports
sudo exportfs -arv

# Verify exported shares
sudo exportfs -v

# Start and enable NFS service
sudo systemctl enable nfs-server
sudo systemctl start nfs-server
Tip: After editing /etc/exports, run sudo exportfs -ra to re-export all shares without restarting the service.

3. Cross-Platform Client Access

Linux Clients

Mount Samba share:

# Install client
sudo apt install cifs-utils

# Mount manually
sudo mount -t cifs //192.168.1.10/Shared /mnt/shared -o username=alice,vers=3.0

# Or via /etc/fstab
//192.168.1.10/Shared /mnt/shared cifs username=alice,password=secret,vers=3.0,uid=1000,gid=1000 0 0

Mount NFS share:

# Install NFS client
sudo apt install nfs-common

# Mount manually
sudo mount -t nfs 192.168.1.10:/srv/nfs/shared /mnt/nfs_shared

# Or via /etc/fstab
192.168.1.10:/srv/nfs/shared /mnt/nfs_shared nfs defaults,noatime 0 0

Windows Clients

Connect to Samba share:

# Via File Explorer
\\192.168.1.10\Shared

# Via command line
net use Z: \\192.168.1.10\Shared /user:alice

# Mount persistently
net use Z: \\192.168.1.10\Shared /user:alice * /p:yes

NFS on Windows:

Windows supports NFS via "Services for NFS" (available in Professional/Enterprise editions):

# Install NFS client (Admin PowerShell)
Install-WindowsFeature -Name NFS-Client

# Mount share
mount -o anon \\192.168.1.10\srv\nfs\shared Z:

macOS Clients

macOS includes built-in SMB and NFS support:

# Connect to Samba (Finder → Go → Connect to Server)
smb://192.168.1.10/Shared

# Mount NFS
sudo mount -t nfs 192.168.1.10:/srv/nfs/shared /Volumes/nfs_shared

# Auto-mount via /etc/fstab (or /etc/auto_master)
192.168.1.10:/srv/nfs/shared /Volumes/nfs_shared nfs rw,noauto 0 0

4. Security Considerations

  • Restrict access by IP: Use firewall rules or hosts allow in smb.conf and subnet masks in /etc/exports.
  • Enable SMB encryption: Set smb encrypt = required and server signing = required in the [global] section to protect against man-in-the-middle attacks.
  • Use strong passwords: Samba passwords are separate from system passwords — enforce complexity with smb.conf policies.
  • Disable guest access: Set map to guest = never if anonymous access is not required.
  • NFSv4 with Kerberos: For production, use NFSv4 with sec=krb5 for encrypted authentication.
  • Firewall rules: Allow only necessary ports: 
    Samba: 137-138/udp, 139/tcp, 445/tcp
NFS:   111/tcp+udp, 2049/tcp
  • Use no_root_squash sparingly: Avoid this option unless absolutely needed; prefer root_squash (default) to map root to nobody.

5. Troubleshooting Common Issues

Samba Issues

ProblemSolution
Cannot connect from WindowsEnable SMB1/CIFS in Windows Features (not recommended) or min protocol = NT1 in smb.conf as last resort. Prefer SMB3.
Permission deniedCheck valid users, filesystem permissions, and SELinux: sudo setsebool -P samba_export_all_rw 1
Slow transfersSet socket options = TCP_NODELAY IPTOS_LOWDELAY in [global]
Authentication failsVerify Samba password with sudo smbpasswd -u alice and check security = user is set

NFS Issues

ProblemSolution
Mount hangsCheck if rpcbind is running: sudo systemctl status rpcbind. Verify firewall allows port 111 and 2049.
Permission denied on NFS shareCompare UID/GID on client and server. Use anonuid/anongid in exports or remap with idmapd for NFSv4.
Stale file handleRun sudo umount -l /mnt/point then sudo mount -a. Avoid keeping files open when server restarts.
NFSv4 ACLs not workingEnsure nfs4_getfacl is installed and the filesystem supports extended attributes.
Tip: For NFS issues, check server logs with journalctl -u nfs-server and client logs with tail -f /var/log/messages (or /var/log/syslog on Debian).

Diagnostic Commands

# Samba
smbclient -L //192.168.1.10 -U alice
testparm -v
smbstatus

# NFS
showmount -e 192.168.1.10
rpcinfo -p 192.168.1.10
exportfs -v
