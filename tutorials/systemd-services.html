Creating and managing systemd service units for background processes, daemons, and scheduled tasks.

What is systemd?

systemd is the init system and service manager used by most modern Linux distributions. It is responsible for booting the system, managing services, mounting filesystems, handling sockets, and scheduling tasks.

systemd uses unit files to define resources it manages. Service units (`.service` files) define daemons and background processes. Other unit types include .timer (scheduled tasks), .socket (socket activation), .mount (filesystem mounts), and .target (grouping units).

Despite some controversy in the Linux community, systemd is the default init system on Ubuntu, Fedora, RHEL, Debian, Arch Linux, and most other major distributions.

Service Unit Files

systemd service unit files are plain-text `.service` files that define how to manage a service. They are located in these directories (in order of priority):

/etc/systemd/system/   # System administrator units (highest priority)
/run/systemd/system/   # Runtime units
/usr/lib/systemd/system/ # Distribution-provided units (lowest priority)

# User services (run as non-root user):
~/.config/systemd/user/  # User service units

# Working with units
sudo systemctl daemon-reload    # Reload unit files after changes
sudo systemctl enable myapp.service # Enable on boot
sudo systemctl disable myapp.service
sudo systemctl start myapp.service
sudo systemctl stop myapp.service
sudo systemctl restart myapp.service
sudo systemctl status myapp.service
systemctl list-units --type=service # List all services
Tip: Always run sudo systemctl daemon-reload after creating or modifying any unit file. systemd caches unit definitions.

Basic Service Structure

A service unit file has three main sections: [Unit], [Service], and [Install]:

[Unit]
Description=My Python Web App
Documentation=https://example.com/docs
After=network.target postgresql.service
Requires=postgresql.service

[Service]
Type=simple
User=myapp
Group=myapp
WorkingDirectory=/opt/myapp
ExecStart=/usr/bin/python3 /opt/myapp/app.py
ExecReload=/bin/kill -HUP $MAINPID
Restart=on-failure
RestartSec=5
Environment=NODE_ENV=production
EnvironmentFile=/etc/myapp/env.conf

[Install]
WantedBy=multi-user.target

The [Unit] section describes the unit and its dependencies. [Service] defines how to start, stop, and restart the process. [Install] specifies when the service should be enabled.

Service Types

The Type= directive tells systemd how the service starts:

TypeDescription
simpleMain process is ExecStart. systemd considers the service started immediately.
forkingProcess forks and the parent exits. systemd tracks the child PID via PIDFile=.
oneshotService runs a one-time task and exits. Often used with RemainAfterExit=yes.
dbusService registers on D-Bus. systemd waits for the bus name to appear.
notifyService sends sd_notify(READY=1) when ready. systemd waits for notification.
idleService starts after all other jobs are dispatched.
# Oneshot example (runs a script at boot)
[Unit]
Description=Setup firewall rules

[Service]
Type=oneshot
RemainAfterExit=yes
ExecStart=/usr/local/bin/setup-firewall.sh
ExecStop=/usr/local/bin/clear-firewall.sh

[Install]
WantedBy=multi-user.target

# Forking example (traditional daemon)
[Service]
Type=forking
PIDFile=/run/myservice.pid
ExecStart=/usr/sbin/myservice
ExecStop=/bin/kill -TERM $MAINPID

Dependencies & Ordering

Control when your service starts relative to other services:

[Unit]
Description=My Web App
After=network.target postgresql.service redis.service
Requires=postgresql.service redis.service
Wants=nginx.service
BindsTo=postgresql.service
PartOf=web-stack.target

# After=    : Start AFTER these units (ordering only)
# Requires=   : Hard dependency — if unit fails, this fails
# Wants=    : Soft dependency — starts if possible, no failure if not
# BindsTo=   : If bound unit stops, this stops too
# PartOf=    : If the target stops/restarts, so does this
# Before=    : Start BEFORE the listed units

# Example: start after PostgreSQL, fail if DB fails
[Unit]
Description=API Server
After=postgresql.service network-online.target
Requires=postgresql.service
Wants=network-online.target

Environment & Variables

Configure environment variables for your service:

[Service]
# Inline environment variables
Environment=PORT=8080
Environment=LOG_LEVEL=info
Environment=DB_HOST=localhost
Environment=NODE_ENV=production

# Load from file (key=value pairs, one per line)
EnvironmentFile=/etc/myapp/env.conf
EnvironmentFile=-/etc/myapp/local-overrides.conf
# The - prefix means "ignore if file doesn't exist"

# For many variables, use a dedicated env file:
# /etc/myapp/env.conf:
# DATABASE_URL=postgres://user:pass@localhost/mydb
# REDIS_URL=redis://localhost:6379
# SECRET_KEY=changeme
# PORT=3000

# Working directory
WorkingDirectory=/opt/myapp

# User and group
User=myappuser
Group=myappgroup

# Set ulimit values
LimitNOFILE=65536
LimitNPROC=4096

# Set umask
UMask=0022

Restart Policies

Control how systemd handles service failures:

[Service]
# Restart when the process exits
Restart=on-failure
RestartSec=5    # Wait 5 seconds between restarts
StartLimitBurst=3 # Max restarts in StartLimitInterval
StartLimitInterval=60s # Reset counter after 60 seconds

# Restart policies:
# no       : Never restart (default)
# on-success   : Restart only on clean exit code (0)
# on-failure   : Restart on non-zero exit, signal, timeout
# on-abnormal   : Restart on signals, timeout, watchdog
# on-watchdog   : Restart on watchdog timeout
# on-abort    : Restart only on uncaught signal
# always     : Always restart regardless of exit status

# Restart limits prevent infinite restart loops
# Use StartLimitInterval and StartLimitBurst

# Watchdog: systemd monitors service health
WatchdogSec=30
# Service must call sd_notify(WATCHDOG=1) every 30 seconds

Logging & journalctl

systemd services log to the journal, accessible via journalctl:

# View logs for a specific service
journalctl -u myapp.service

# Follow logs in real-time
journalctl -u myapp.service -f

# Show last 50 lines
journalctl -u myapp.service -n 50

# Show logs since last boot
journalctl -u myapp.service -b

# Show logs for a time range
journalctl -u myapp.service --since "2024-01-01" --until "2024-01-02"
journalctl -u myapp.service --since "1 hour ago"

# Show only errors
journalctl -u myapp.service -p err

# Output formats
journalctl -u myapp.service -o json-pretty
journalctl -u myapp.service -o verbose

# Show logs from the current boot only
journalctl -b

# Clear old logs (keep last 500 MB)
sudo journalctl --vacuum-size=500M

# Keep logs for 2 weeks
sudo journalctl --vacuum-time=2weeks

# Control log output from the service itself
[Service]
StandardOutput=journal
StandardError=journal
# Or redirect to syslog
SyslogIdentifier=myapp

Timers

systemd timers replace cron as a modern scheduler. Timer files pair with a service unit:

# /etc/systemd/system/backup.service
[Unit]
Description=Daily backup service

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh
User=backupuser

# /etc/systemd/system/backup.timer
[Unit]
Description=Run backup daily at 2 AM

[Timer]
OnCalendar=daily
OnCalendar=*-*-* 02:00:00
Persistent=true  # Run immediately if missed (e.g., after boot)
RandomizedDelaySec=300 # Random delay to avoid thundering herd
Unit=backup.service

[Install]
WantedBy=timers.target

# Enable the timer (not the service!)
sudo systemctl enable backup.timer --now
sudo systemctl list-timers --all

# OnCalendar examples:
# OnCalendar=daily
# OnCalendar=hourly
# OnCalendar=Mon..Fri 09:00:00
# OnCalendar=*-*-01/14:00:00 # 1st and 15th of each month
# OnCalendar=Sun 03:00:00

# Timer vs cron: timers support
# - Randomized delays
# - Persistent (catch up after downtime)
# - Dependencies on systemd units
# - Accurate to microsecond precision
# - Built-in logging via journald

Troubleshooting Services

Common issues and debugging techniques for systemd services:

  • Check status first: sudo systemctl status myservice shows whether it's running, the last few log lines, and the PID.
  • View full logs: journalctl -u myservice -n 100 --no-pager to see the last 100 log entries.
  • Validate unit file: systemd-analyze verify /etc/systemd/system/myservice.service catches syntax errors.
  • Check dependencies: systemctl list-dependencies myservice shows the dependency tree.
  • Permission denied: Ensure the User= specified has read/execute access to ExecStart and the working directory.
  • Service won't start: Check journalctl -xe for the "failed to start" reason. Common causes: missing env files, wrong paths, port in use.
  • Restart loop: Check Restart= policy and StartLimitInterval. The service may be crashing immediately.
  • Environment variables not loading: Verify EnvironmentFile= path and format. Each line must be KEY=VALUE without spaces around =.
  • Analyze boot performance: systemd-analyze blame shows which services take longest to start.
