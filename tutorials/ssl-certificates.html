Understand TLS, get free certificates with Let's Encrypt, and secure your web services properly.

What is SSL / TLS?

TLS (Transport Layer Security) is the cryptographic protocol that encrypts data between a client and a server. It is the successor to SSL (Secure Sockets Layer), which is now deprecated. People still commonly say "SSL" when they mean TLS.

TLS provides three critical guarantees:

  • Encryption — data cannot be read by third parties in transit
  • Authentication — you are communicating with the real server, not an impostor
  • Integrity — data cannot be modified in transit without detection

How Certificates Work

A TLS certificate is a digital document that binds a public key to an identity (domain name). It is signed by a Certificate Authority (CA) that vouches for the authenticity of the certificate.

The handshake process:

  1. Client connects to server and requests a secure connection
  2. Server sends its certificate (containing the public key)
  3. Client checks the certificate against its trusted CA store
  4. If valid, client generates a session key, encrypts it with the server's public key, and sends it
  5. Both sides use the session key for symmetric encryption

Let's Encrypt & Certbot

Let's Encrypt is a free, automated, and open CA run by the Internet Security Research Group (ISRG). It provides 90-day certificates that can be auto-renewed. Certbot is the official client tool by the Electronic Frontier Foundation (EFF).

# Install Certbot with Nginx plugin (Ubuntu/Debian)
sudo apt update
sudo apt install certbot python3-certbot-nginx

# Install Certbot with Apache plugin
sudo apt install certbot python3-certbot-apache

# Test on other systems
sudo dnf install certbot python3-certbot-nginx # Fedora
sudo pacman -S certbot certbot-nginx       # Arch

Obtaining Certificates

Certbot automatically obtains and installs certificates:

# Get a certificate (Nginx plugin — auto-configures HTTPS)
sudo certbot --nginx -d example.com -d www.example.com

# Get a certificate (Apache plugin)
sudo certbot --apache -d example.com

# Get a certificate (standalone — temporary web server on port 80)
sudo certbot certonly --standalone -d example.com

# Get a wildcard certificate (requires DNS challenge)
sudo certbot certonly --manual --preferred-challenges dns -d *.example.com

Certificates are stored in /etc/letsencrypt/live/example.com/:

  • fullchain.pem — Your certificate + CA chain (use this in web servers)
  • privkey.pem — Your private key (keep this secret!)
  • chain.pem — CA chain only
  • cert.pem — Your certificate only

Auto-Renewal

Certbot creates a systemd timer that checks for renewal twice daily:

# Check the timer status
sudo systemctl status certbot.timer

# Test renewal (dry run)
sudo certbot renew --dry-run

# Force renewal (if a cert is expiring soon)
sudo certbot renew

# Manual renewal for all certificates
sudo certbot renew --force-renewal

If you used the --nginx or --apache plugin, renewal will automatically reload the web server. For standalone setups, add a --pre-hook and --post-hook to stop/start your server.

Nginx HTTPS Configuration

Production-grade HTTPS server block:

server {
  listen 443 ssl http2;
  server_name example.com;

  # Certificate paths
  ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
  ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

  # Modern TLS config (as of 2025)
  ssl_protocols TLSv1.2 TLSv1.3;
  ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
  ssl_prefer_server_ciphers off;
  ssl_session_cache shared:SSL:10m;
  ssl_session_timeout 1d;
  ssl_session_tickets off;

  # HSTS (uncomment after confirming everything works)
  # add_header Strict-Transport-Security "max-age=63072000" always;

  root /var/www/example.com;
  index index.html;

  location / {
    try_files $uri $uri/ =404;
  }
}

server {
  listen 80;
  server_name example.com www.example.com;
  return 301 https://$server_name$request_uri;
}

Self-Signed Certificates

For internal/testing use where you don't need a CA:

# Generate a self-signed certificate (valid for 365 days)
sudo openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout /etc/ssl/private/selfsigned.key \
  -out /etc/ssl/certs/selfsigned.crt \
  -subj "/C=US/ST=State/L=City/O=Organization/CN=myserver.local"

# Generate with SAN (Subject Alternative Name) — needed for modern browsers
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout selfsigned.key -out selfsigned.crt \
  -subj "/CN=myserver.local" \
  -addext "subjectAltName=DNS:myserver.local,DNS:localhost,IP:192.168.1.100"

Self-signed certificates will show a browser warning — they are not trusted by default. Add them to your system's trust store for local use.

Certificate Formats & Conversion

Different platforms need different formats:

FormatExtensionUse Case
PEM (Base64).pem, .crt, .keyLinux, Nginx, Apache, HAProxy
PKCS12.p12, .pfxWindows, IIS, Java keystores
DER (Binary).derJava, some embedded devices
JKS.jksJava KeyStore

Convert between formats with OpenSSL:

# PEM to PKCS12
openssl pkcs12 -export -out cert.p12 -inkey privkey.pem -in cert.pem -certfile chain.pem

# PEM to DER
openssl x509 -in cert.pem -outform der -out cert.der

# PKCS12 to PEM
openssl pkcs12 -in cert.p12 -out cert.pem -nodes

Testing & Validation

Tools to verify your SSL setup:

# Check certificate details
openssl x509 -in /etc/letsencrypt/live/example.com/cert.pem -text -noout

# Test SSL connection
openssl s_client -connect example.com:443 -tls1_3

# Scan for vulnerabilities
sudo apt install testssl.sh
testssl.sh https://example.com

Online tools: SSL Labs (ssllabs.com/ssltest) and SSL Checker (sslshopper.com) provide comprehensive analysis.

Common Issues

  • Certificate expired: Run sudo certbot renew and check the systemd timer is active.
  • Domain mismatch: Ensure the certificate covers all domain variants (example.com + www.example.com).
  • Port 80 blocked: Let's Encrypt's HTTP-01 challenge requires port 80 (or use DNS challenge with --preferred-challenges dns).
  • Certificate chain incomplete: Always use fullchain.pem, not cert.pem alone.
  • Mixed content: All resources (images, scripts, styles) must load over HTTPS — use relative URLs or upgrade insecure requests.
