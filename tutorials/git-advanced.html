Interactive Rebase

# Squash, reword, reorder the last 3 commits
git rebase -i HEAD~3

# Commands inside editor:
# pick  = use commit
# reword = change message only
# squash = merge into previous commit
# fixup  = squash but discard message
# drop  = remove commit
# edit  = stop to amend
Never rebase commits that have already been pushed to a shared branch. Use --force-with-lease instead of --force to avoid overwriting others' work.

Cherry-Pick

# Apply a specific commit onto current branch
git cherry-pick abc123

# Cherry-pick a range
git cherry-pick abc123..def456

# Cherry-pick without committing (edit first)
git cherry-pick -n abc123

Reflog — Your Safety Net

# View all HEAD movements
git reflog

# Recover a "lost" commit after reset
git reset --hard HEAD@{2}

# Recover after rebase gone wrong
git reflog
git checkout -b recovery HEAD@{5}

# Log with relative time
git reflog --date=relative
Reflog is local-only and expires entries after 90 days. It's your parachute when a rebase or reset goes wrong.

Bisect — Find the Bug

# Start bisect
git bisect start
git bisect bad     # current commit is broken
git bisect good v1.0  # v1.0 was working

# Git checks out mid-point. Mark each:
git bisect good # or
git bisect bad

# When done:
git bisect reset

# Automate with a script
git bisect start HEAD v1.0 --
git bisect run npm test

Git Hooks

# .git/hooks/pre-commit
npm run lint || { echo "Lint failed"; exit 1; }

# .git/hooks/commit-msg
grep -qE "^(feat|fix|chore|docs|refactor)\(.*\):" "$1" \
 || { echo "Conventional commit format required"; exit 1; }

Submodules

# Add a submodule
git submodule add https://github.com/example/lib.git lib/

# Clone with submodules
git clone --recurse-submodules https://github.com/user/repo.git

# Update all submodules
git submodule update --remote --merge

# Remove a submodule
git submodule deinit lib/
git rm lib/
rm -rf .git/modules/lib/

Workflow Strategies

Git Flow

# Branches: main, develop, feature/*, release/*, hotfix/*
git flow init
git flow feature start new-feature
git flow feature finish new-feature

Trunk-Based (simpler)

# Short-lived feature branches, rebase onto main
git checkout -b feat/my-feature
git fetch origin
git rebase origin/main
# PR — merge with squash, then delete branch
Trunk-based with short-lived branches and squash-merging keeps history linear and readable. Git Flow suits release-cycle projects with versioned deployments.
