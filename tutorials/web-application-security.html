Web Application Security

OWASP Top 10, XSS, CSRF, SQL injection, and securing web applications.

OWASP Top 10 Deep Dive

The OWASP Top 10 is the standard awareness document for web application security. Current (2021): A01 — Broken Access Control (95%+ apps tested), A02 — Cryptographic Failures, A03 — Injection (SQL, NoSQL, OS, LDAP), A04 — Insecure Design, A05 — Security Misconfiguration, A06 — Vulnerable Components, A07 — Auth Failures, A08 — Data Integrity, A09 — Logging/Monitoring, A10 — SSRF.

Most critical: Broken Access Control moved from #5 to #1 in 2021, reflecting the shift to APIs and microservices where access control is more complex and frequently broken.

Cross-Site Scripting (XSS)

XSS injects malicious scripts into web pages viewed by others. Types: Stored (persistent — saved in database, affects all visitors), Reflected (non-persistent — in URL/request, user must click), DOM-based (vulnerability in client-side JavaScript).

Prevention: context-aware output encoding (HTML entity encode for HTML body, JavaScript encode for JS context), Content Security Policy (CSP — restricts which scripts can execute, inline-script blocking), input validation (whitelist, not blacklist), and HTTP-only cookies for session tokens.

SQL Injection Prevention

SQL injection occurs when user input is directly concatenated into SQL queries. Attackers can read, modify, or delete database contents. Prevention is straightforward: always use parameterized queries (prepared statements) — never concatenate inputs into SQL strings.

Other defenses: ORMs (TypeORM, Prisma, SQLAlchemy — they use parameterized queries internally), stored procedures (less flexible), input validation (type checking, length limits, pattern matching), least privilege DB accounts (SELECT-only for read operations), and WAF rules (mod_security) for layered defense.

Authentication & Session Security

Secure authentication requires: strong password policies (12+ chars, bcrypt/argon2 hashing, work factor 10+, COST=12), MFA (TOTP, SMS backup, WebAuthn/passkeys), account lockout (10 failed attempts = 15-min lock), and rate limiting on login endpoints.

Session security: HTTP-only, Secure, SameSite=lax/strict cookies; short session expiry (15 min inactivity); session rotation after login; bind sessions to IP + User-Agent (watch for changes); token-based auth (JWT) with short expiry (15 min) and refresh tokens (7 days, rotated on use).

API Security Best Practices

API security: authentication (JWT, OAuth 2.0, API keys — depending on use case), rate limiting (100 req/min per user, 1000 req/min per IP), input validation (JSON schema validation, type checking, whitelist allowed values), TLS 1.2+ for all endpoints, and CORS (strict origin whitelist, avoid wildcards).

Additional: API versioning (v1, v2 headers/URLs), idempotency keys (POST retry safety), request logging (with sensitive data masking), webhook secret verification (HMAC signatures), GraphQL depth limiting / query cost analysis, and OWASP API Security Top 10.
