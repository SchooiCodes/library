📊 Server Monitoring with Prometheus & Grafana

Monitor server metrics, visualize data, and set up alerts using Prometheus and Grafana.

What is Monitoring?

Server monitoring is the practice of collecting, analyzing, and visualizing metrics about your infrastructure to ensure reliability, performance, and availability. It answers questions like: Is my server up? How much CPU is being used? Is disk space running low? Are requests taking too long?

Prometheus is an open-source systems monitoring and alerting toolkit originally built at SoundCloud. It collects metrics from configured targets at given intervals, evaluates rule expressions, displays results, and triggers alerts.

Grafana is a multi-platform analytics and interactive visualization web application. It connects to Prometheus (and many other data sources) to create dashboards, graphs, and alerts.

Prometheus Overview

Prometheus uses a pull model — it scrapes metrics from HTTP endpoints on monitored targets. Key concepts:

  • Metrics — Numeric measurements collected over time, identified by metric name and labels.
  • Labels — Key-value pairs that add dimensionality (e.g., instance="server1", job="node").
  • Targets — Endpoints Prometheus scrapes, defined in prometheus.yml.
  • Alertmanager — Handles alerts deduplication, grouping, and routing to notification channels.
  • Time series database — Stores all scraped metrics with timestamps.
# A Prometheus time series looks like:
node_cpu_seconds_total{cpu="0",mode="user",instance="server1:9100"} 12345.67

Installing Prometheus

Install Prometheus from the official website or package manager:

# Download and extract
wget https://github.com/prometheus/prometheus/releases/latest/download/prometheus-*.tar.gz
tar xvf prometheus-*.tar.gz
cd prometheus-*

# Create user
sudo useradd --no-create-home --shell /bin/false prometheus
sudo mkdir /etc/prometheus /var/lib/prometheus

# Copy binaries
sudo cp prometheus promtool /usr/local/bin/
sudo cp -r consoles console_libraries /etc/prometheus/

# Basic prometheus.yml
cat > /etc/prometheus/prometheus.yml << 'PEOF'
global:
 scrape_interval: 15s
 evaluation_interval: 15s

scrape_configs:
 - job_name: 'prometheus'
  static_configs:
   - targets: ['localhost:9090']

 - job_name: 'node'
  static_configs:
   - targets: ['localhost:9100']
PEOF
Tip: Access the Prometheus web UI at http://localhost:9090. Use systemctl status prometheus to check the service.

Exporters (Node Exporter)

Exporters are agents that collect system metrics and expose them in Prometheus format. Node Exporter provides hardware and OS metrics for Linux servers:

# Install Node Exporter
wget https://github.com/prometheus/node_exporter/releases/latest/download/node_exporter-*.tar.gz
tar xvf node_exporter-*.tar.gz
sudo cp node_exporter /usr/local/bin/

# Create systemd service for Node Exporter
cat > /etc/systemd/system/node_exporter.service << 'NEOF'
[Unit]
Description=Node Exporter
After=network.target

[Service]
User=node_exporter
Group=node_exporter
Type=simple
ExecStart=/usr/local/bin/node_exporter \
  --collector.logind \
  --collector.systemd \
  --collector.processes \
  --web.listen-address=:9100

[Install]
WantedBy=multi-user.target
NEOF

sudo systemctl enable node_exporter --now
# Metrics available at http://localhost:9100/metrics

# Other useful exporters:
# - blackbox_exporter: HTTP/HTTPS/DNS/TCP probing
# - mysqld_exporter: MySQL/MariaDB metrics
# - postgres_exporter: PostgreSQL metrics
# - nginx_exporter: Nginx metrics via stub_status
# - redis_exporter: Redis metrics

PromQL Basics

PromQL (Prometheus Query Language) is used to query time series data. Here are essential queries:

# Simple metric query
node_cpu_seconds_total

# With label filtering
node_cpu_seconds_total{cpu="0", mode="user"}
node_load1{instance="server1:9100"}

# Rate of change per second (for counters)
rate(node_cpu_seconds_total{mode="user"}[5m])

# Average CPU usage by instance
avg(rate(node_cpu_seconds_total{mode="user"}[5m])) by (instance)

# Memory usage percentage
(1 - node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes) * 100

# Disk space available percentage
node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"} * 100

# Top 5 processes by memory
topk(5, node_processes_memory_rss)

# Aggregation functions
sum(), avg(), min(), max(), count(), quantile()
rate(), irate(), increase() # For counters
histogram_quantile()     # For histograms

# Example: 95th percentile of request duration
histogram_quantile(0.95, sum(rate(http_request_duration_seconds_bucket[5m])) by (le))

Grafana Overview

Grafana is the visualization layer on top of Prometheus. It provides:

  • Interactive dashboards — Drag-and-drop panel builder with graphs, tables, heatmaps, and more.
  • Data sources — Prometheus, InfluxDB, Elasticsearch, MySQL, PostgreSQL, CloudWatch, and 50+ others.
  • Alerting — Built-in alert rules with notifications to Slack, PagerDuty, email, webhooks, and more.
  • Annotations — Overlay events (deployments, incidents) on graphs.
  • Variables — Dynamic dashboards with template variables for switching between instances, jobs, or time ranges.
  • Explore mode — Ad-hoc query interface for investigating metrics without building a dashboard.

Installing Grafana

Install Grafana using the official APT repository or as a Docker container:

# Debian / Ubuntu
sudo apt-get install -y software-properties-common wget
sudo wget -q -O /usr/share/keyrings/grafana.key https://apt.grafana.com/gpg.key
echo "deb [signed-by=/usr/share/keyrings/grafana.key] https://apt.grafana.com stable main" | sudo tee /etc/apt/sources.list.d/grafana.list
sudo apt-get update
sudo apt-get install grafana

# Start Grafana
sudo systemctl enable grafana-server --now

# Docker
docker run -d -p 3000:3000 --name=grafana grafana/grafana-oss

# Default login: admin / admin
# Access at: http://localhost:3000

# Add Prometheus as a data source:
# Configuration > Data Sources > Add data source > Prometheus
# URL: http://localhost:9090
# Save & Test

Alerting Rules

Prometheus evaluates alerting rules and sends notifications through Alertmanager:

# /etc/prometheus/rules.yml
groups:
 - name: server_alerts
  rules:
   - alert: InstanceDown
    expr: up == 0
    for: 5m
    labels:
     severity: critical
    annotations:
     summary: "Instance {{ $labels.instance }} down"
     description: "{{ $labels.instance }} has been down for more than 5 minutes."

   - alert: HighCPUUsage
    expr: 100 - (avg by(instance)(rate(node_cpu_seconds_total{mode="idle"}[5m])) * 100) > 80
    for: 10m
    labels:
     severity: warning
    annotations:
     summary: "High CPU usage on {{ $labels.instance }}"
     description: "CPU usage is {{ $value | humanizePercentage }} on {{ $labels.instance }}."

   - alert: DiskSpaceLow
    expr: (node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"}) * 100 < 10
    for: 5m
    labels:
     severity: critical
    annotations:
     summary: "Disk space low on {{ $labels.instance }}"
     description: "Only {{ $value | humanizePercentage }} disk space remaining."

# Add rules to prometheus.yml:
rule_files:
 - "rules.yml"

# Configure Alertmanager to send notifications (e.g., Slack)
# /etc/alertmanager/alertmanager.yml
route:
 receiver: 'slack'
receivers:
 - name: 'slack'
  slack_configs:
   - api_url: 'https://hooks.slack.com/services/YOUR/WEBHOOK'
    channel: '#alerts'
    send_resolved: true

Best Practices

  • Use meaningful metric names following the naming convention: namespace_job_unit (e.g., node_cpu_seconds_total).
  • Label carefully — too many label values creates high cardinality and performance issues. Avoid labels with unlimited values (user IDs, email addresses).
  • Set appropriate scrape intervals — 15s is standard. Go higher (30-60s) for non-critical metrics to reduce load.
  • Use recording rules for expensive queries that run frequently. Pre-compute them as new time series.
  • Monitor the monitor — set up alerts on Prometheus itself. If it goes down, you lose visibility.
  • Use Grafana variables to make dashboards reusable across instances, environments, and teams.
  • Set retention limits — configure --storage.tsdb.retention.time in Prometheus to control storage usage.
  • Backup dashboards — export Grafana dashboards as JSON and store them in version control (Git).
