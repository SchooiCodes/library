What is PostgreSQL?

PostgreSQL (often called Postgres) is a powerful, open-source object-relational database system with over 35 years of active development. It is known for its reliability, feature robustness, and performance.

PostgreSQL supports advanced data types, full-text search, JSON/JSONB for NoSQL-style document storage, custom functions, triggers, and concurrent transactions with full ACID compliance. It is extensible — you can define your own data types, operators, and index methods.

Major companies including Apple, Reddit, Instagram, and Twitch use PostgreSQL in production. It consistently ranks as the most loved database in developer surveys.

Installation & Setup

Install PostgreSQL using your package manager:

# Debian / Ubuntu
sudo apt update
sudo apt install postgresql postgresql-contrib

# RHEL / Fedora
sudo dnf install postgresql-server postgresql-contrib
sudo postgresql-setup --initdb

# macOS (Homebrew)
brew install postgresql@15

# Start the service
sudo systemctl enable postgresql --now

# Switch to the postgres user
sudo -i -u postgres
psql

After installation, you can access the PostgreSQL interactive terminal with psql. The default superuser is postgres. Set a password with \password postgres inside psql.

Tip: Use pg_lsclusters to list all PostgreSQL clusters on your system. Default port is 5432.

Basic SQL

PostgreSQL uses standard SQL with some extensions. Here are essential commands:

-- Create a database
CREATE DATABASE mydb;

-- Connect to it
\c mydb

-- Create a table
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(255) UNIQUE NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Insert data
INSERT INTO users (name, email) VALUES
  ('Alice', 'alice@example.com'),
  ('Bob', 'bob@example.com');

-- Query data
SELECT * FROM users WHERE name = 'Alice';

-- Update
UPDATE users SET email = 'alice@newdomain.com' WHERE id = 1;

-- Delete
DELETE FROM users WHERE id = 2;

Use \dt to list tables, \d tablename to describe a table, and \? for psql help.

Creating Tables

PostgreSQL offers a rich set of data types for table columns:

CREATE TABLE products (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(200) NOT NULL,
  description TEXT,
  price NUMERIC(10, 2) CHECK (price > 0),
  stock INTEGER DEFAULT 0,
  category_id INTEGER REFERENCES categories(id),
  tags JSONB DEFAULT '[]',
  metadata JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

Common data types: INTEGER, BIGINT, NUMERIC(p,s) (exact decimal), VARCHAR(n), TEXT, BOOLEAN, DATE, TIMESTAMP, TIMESTAMPTZ (timezone-aware), JSON, JSONB (indexable binary JSON), UUID, BYTEA (binary), and ARRAY.

Constraints & Indexes

Constraints enforce data integrity. Indexes speed up queries:

-- Constraints
CREATE TABLE orders (
  id SERIAL PRIMARY KEY,
  user_id INTEGER NOT NULL REFERENCES users(id),
  status VARCHAR(20) CHECK (status IN ('pending', 'shipped', 'delivered')),
  total NUMERIC(10,2) NOT NULL,
  UNIQUE (user_id, order_date),
  EXCLUDE USING gist (period WITH &&) -- No overlapping time periods
);

-- Indexes
CREATE INDEX idx_orders_user_id ON orders(user_id);
CREATE INDEX idx_orders_status_date ON orders(status, created_at DESC);
CREATE UNIQUE INDEX idx_users_email ON users(email);
CREATE INDEX idx_products_tags ON products USING gin(tags); -- JSONB index
CREATE INDEX idx_products_search ON products USING gin(to_tsvector('english', name || ' ' || description));

Use EXPLAIN ANALYZE before a query to see if your indexes are being used. Over-indexing can slow down writes, so index strategically.

Joins & Relations

PostgreSQL fully supports relational joins between tables:

-- Inner join with multiple tables
SELECT
  u.name,
  o.id AS order_id,
  o.total,
  p.name AS product_name,
  oi.quantity
FROM users u
JOIN orders o ON o.user_id = u.id
JOIN order_items oi ON oi.order_id = o.id
JOIN products p ON p.id = oi.product_id
WHERE u.email = 'alice@example.com'
ORDER BY o.created_at DESC;

-- Left join (users with or without orders)
SELECT u.name, COUNT(o.id) AS order_count
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id;

-- CTE (Common Table Expression) for complex queries
WITH popular_products AS (
  SELECT product_id, SUM(quantity) AS total_sold
  FROM order_items
  GROUP BY product_id
  ORDER BY total_sold DESC
  LIMIT 10
)
SELECT p.*, pp.total_sold
FROM products p
JOIN popular_products pp ON pp.product_id = p.id;

PostgreSQL supports INNER, LEFT, RIGHT, FULL OUTER, CROSS, and NATURAL joins, plus LATERAL subqueries for advanced row-by-row processing.

Views & Functions

Views are saved queries you can treat as virtual tables. Functions encapsulate logic on the server side:

-- View for active orders
CREATE VIEW active_orders AS
SELECT o.*, u.name AS user_name
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.status NOT IN ('delivered', 'cancelled');

-- Function to calculate order total
CREATE FUNCTION calculate_order_total(order_id INTEGER)
RETURNS NUMERIC(10,2) AS $$
DECLARE
  total NUMERIC(10,2);
BEGIN
  SELECT SUM(p.price * oi.quantity) INTO total
  FROM order_items oi
  JOIN products p ON p.id = oi.product_id
  WHERE oi.order_id = order_id;
  RETURN total;
END;
$$ LANGUAGE plpgsql;

PostgreSQL supports stored procedures (with CREATE PROCEDURE), triggers (with CREATE TRIGGER), and window functions for analytical queries using OVER (PARTITION BY ...).

Backup & Restore

Use pg_dump and pg_restore for database backup and recovery:

# Backup a single database
pg_dump mydb > mydb_backup.sql

# Backup with custom format (compressed, parallel restores)
pg_dump -Fc mydb > mydb_backup.dump

# Backup all databases
pg_dumpall > all_dbs.sql

# Restore from SQL dump
psql mydb < mydb_backup.sql

# Restore from custom format
pg_restore -d mydb mydb_backup.dump

# Continuous archiving (WAL) for point-in-time recovery
# Edit postgresql.conf:
# wal_level = replica
# archive_mode = on
# archive_command = 'cp %p /backups/%f'

For production, use pg_dump -Fc combined with cron jobs. For large databases, consider pg_dump --jobs=N for parallel dumps or PostgreSQL's built-in streaming replication.

Performance Tuning

Key configuration parameters in postgresql.conf for tuning:

ParameterDescriptionSuggested Value
shared_buffersMemory for caching data25% of RAM
work_memMemory per sort/hash operation4-16 MB
maintenance_work_memMemory for VACUUM, CREATE INDEX10% of RAM
effective_cache_sizeOS file cache estimate75% of RAM
random_page_costCost of random I/O1.1 (SSD) / 4.0 (HDD)
max_connectionsConcurrent connections20-100 (use pgbouncer for more)

Run VACUUM ANALYZE regularly to update statistics and reclaim storage. Use pg_stat_statements extension to identify slow queries.

User Management

PostgreSQL uses a role-based access control system:

-- Create a user with login
CREATE USER app_user WITH PASSWORD 'secure_password';

-- Create a read-only role
CREATE ROLE read_only;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO read_only;

-- Grant schema-level permissions
GRANT USAGE, CREATE ON SCHEMA public TO app_user;
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO app_user;
ALTER DEFAULT PRIVILEGES IN SCHEMA public GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO app_user;

-- Create group role and add user
CREATE ROLE developers;
GRANT developers TO app_user;

-- Revoke permissions
REVOKE DELETE ON orders FROM app_user;

Use pg_hba.conf to control client authentication (md5, scram-sha-256, trust, peer) and connection rules per IP range. Never use trust in production.
