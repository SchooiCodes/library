What is Git?

Git is a distributed version control system that tracks changes to files over time. Created by Linus Torvalds in 2005 to manage the Linux kernel source code, it has become the industry standard for software development.

With Git you can:

  • Track every change made to your project
  • Revert to any previous state
  • Work on multiple features simultaneously using branches
  • Collaborate with other developers without stepping on each other's toes
  • Sync your code with remote repositories like GitHub, GitLab, or Bitbucket

First-Time Setup

Configure Git with your identity and preferences. Only needed once per machine:

# Set your name (appears on commits)
git config --global user.name "Your Name"

# Set your email (must match GitHub/GitLab)
git config --global user.email "your@email.com"

# Set default branch name to 'main'
git config --global init.defaultBranch main

# Enable color output
git config --global color.ui auto

# View all settings
git config --list
Your global Git config is stored in ~/.gitconfig. You can edit it directly with any text editor.

Basic Workflow

The core Git workflow follows these steps. Once you master this cycle, you know 80% of what you'll do day-to-day:

1. Initialize a Repository

# Create a new Git repository in the current folder
git init

# Or clone an existing project
git clone https://github.com/username/repo.git

2. Check Status

# See which files are modified, staged, or untracked
git status

3. Stage Changes

# Stage a specific file
git add file.txt

# Stage all changes in the current directory
git add .

# Stage all changes in the entire repo
git add -A

# Unstage a file (but keep changes)
git restore --staged file.txt

4. Commit Changes

# Commit staged changes with a message
git commit -m "Add user authentication module"

# Add all tracked files and commit in one step
git commit -am "Fix login bug"

# View commit history
git log

# Compact log view
git log --oneline

5. View Changes

# Show unstaged changes (what you've modified)
git diff

# Show staged changes (what will be committed)
git diff --staged

# Show changes in a specific commit
git show abc123

Branching

Branches let you work on features, fixes, or experiments in isolation. The default branch is usually named main (or master on older repos).

Best Practice: Never commit directly to main. Create a feature branch for each new feature or bug fix, then merge it via a pull request.
# List branches (* = current branch)
git branch

# Create a new branch
git branch my-feature

# Switch to a branch
git checkout my-feature       # Older way
git switch my-feature         # Newer way (Git 2.23+)

# Create and switch in one command
git checkout -b my-feature    # Older
git switch -c my-feature      # Newer

# Merge a branch into the current one
git checkout main
git merge my-feature

# Delete a branch (after merging)
git branch -d my-feature

# Force delete an unmerged branch
git branch -D my-feature

Feature Branch Workflow

# Start from main
git checkout main
git pull

# Create feature branch
git checkout -b feature-login

# Work, stage, commit
git add login.html login.css login.js
git commit -m "Add login page"

# Incorporate upstream changes
git checkout main
git pull
git checkout feature-login
git merge main    # or git rebase main

# Push and create PR
git push -u origin feature-login

Working with GitHub

GitHub hosts your Git repositories in the cloud and adds collaboration features on top. Here's how to connect your local repo:

Connect a Local Repo to GitHub

# 1. Create a repository on github.com (don't initialize it)

# 2. Add the remote URL
git remote add origin https://github.com/your-username/repo-name.git

# 3. Push your code
git push -u origin main

# -u sets upstream tracking so future pushes are just 'git push'

Clone an Existing Repo

# Download a repository to your machine
git clone https://github.com/username/repo-name.git

# Clone into a specific folder
git clone https://github.com/username/repo-name.git my-folder

Stay in Sync

# Download remote changes (does NOT merge)
git fetch

# Download AND merge remote changes
git pull

# Push your local commits to the remote
git push

# View remote URLs
git remote -v

# Add another remote (e.g., a fork)
git remote add upstream https://github.com/original-owner/repo.git

# Fetch from upstream
git fetch upstream

Undoing Changes

Made a mistake? Git has several ways to undo changes at different stages:

Scenario Command
Discard unstaged changes in a filegit restore file.txt
Discard all unstaged changesgit restore .
Unstage a file (keep changes)git restore --staged file.txt
Edit the last commit messagegit commit --amend -m "new message"
Add forgotten files to last commitgit add forgotten.txt && git commit --amend --no-edit
Undo last commit (keep changes staged)git reset --soft HEAD~1
Undo last commit (keep changes in working dir)git reset --mixed HEAD~1 (default)
Undo last commit (discard all changes)git reset --hard HEAD~1
Revert a commit (safe — creates new commit)git revert abc123
Warning: git reset --hard permanently discards uncommitted changes. Use with caution! If you've already pushed, prefer git revert instead of reset.

Useful Commands

These commands will make your Git workflow faster and more powerful:

# Temporarily save uncommitted changes
git stash
git stash pop          # Restore and remove stash
git stash list         # List all stashes
git stash apply stash@{0}  # Apply without removing
git stash drop stash@{0}   # Delete a stash

# Visual commit graph
git log --oneline --graph --all

# Show who changed what in a file
git blame file.txt

# Search commits by message
git log --grep="fix"

# Show the branch structure
git log --graph --oneline --decorate

# Undo all changes since a specific commit
git checkout abc123 -- .  # Restore files to that state

# Clean untracked files
git clean -n            # Dry-run (show what would be deleted)
git clean -fd           # Force delete untracked files and folders

# Show differences between branches
git diff main..feature

# Rename a branch
git branch -m old-name new-name

Example Workflow: Contributing to a Project

Here's the complete workflow for contributing to an open-source project on GitHub:

# 1. Fork the repository on GitHub

# 2. Clone your fork
git clone https://github.com/your-username/project.git
cd project

# 3. Add the original repo as upstream
git remote add upstream https://github.com/original-owner/project.git

# 4. Create a feature branch
git checkout -b my-feature

# 5. Make your changes, then stage and commit
git add .
git commit -m "Add my feature"

# 6. Sync with upstream changes
git fetch upstream
git checkout main
git merge upstream/main
git checkout my-feature
git merge main   # Resolve conflicts if any

# 7. Push your branch to your fork
git push -u origin my-feature

# 8. Go to GitHub and create a Pull Request from your fork
# 9. After PR is merged, clean up
git checkout main
git pull upstream main
git branch -d my-feature

GitHub Features

Beyond hosting Git repos, GitHub provides a complete development platform:

Pull Requests (PRs) — Propose changes, review code, and merge branches
Issues — Track bugs, feature requests, and tasks
Actions — CI/CD automation (test, build, deploy)
Pages — Host static websites for free
Discussions — Community conversations and Q&A
Projects — Kanban-style project management
Code Spaces — Cloud development environments (VS Code in browser)
Security — Dependabot alerts, secret scanning, code review
