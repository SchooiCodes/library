What is Ansible?

Ansible is an open-source IT automation tool that handles configuration management, application deployment, cloud provisioning, and orchestration. It was acquired by Red Hat in 2015 and has become the standard for infrastructure-as-code.

Key differentiators from tools like Puppet or Chef:

  • Agentless — no software needs to be installed on managed nodes (uses SSH)
  • Idempotent — running a playbook multiple times produces the same result
  • Declarative YAML — playbooks are human-readable YAML files, not code
  • Push-based — the control node pushes configs to servers (no pull agents)

Installation

Ansible runs on the control node (your laptop or a management server). Managed nodes only need Python and SSH:

# Debian / Ubuntu
sudo apt update
sudo apt install ansible

# RHEL / Fedora
sudo dnf install ansible

# macOS (Homebrew)
brew install ansible

# Verify
ansible --version

Inventory

The inventory defines which hosts Ansible manages. Default location: /etc/ansible/hosts in INI format:

# Simple INI inventory
web1 ansible_host=192.168.1.10
web2 ansible_host=192.168.1.11

[webservers]
web1
web2

[databases]
db1 ansible_host=192.168.1.20

[all:vars]
ansible_user=deploy
ansible_python_interpreter=/usr/bin/python3

You can also use YAML inventory format. Test connectivity with:

ansible all -i inventory -m ping

Ad-Hoc Commands

Run quick tasks without writing a playbook:

# Check uptime on all servers
ansible all -i inventory -a "uptime"

# Install a package
ansible webservers -m apt -a "name=nginx state=present" -b

# Copy a file
ansible all -m copy -a "src=/local/file dest=/remote/path"

# Restart a service
ansible databases -m systemd -a "name=postgresql state=restarted" -b

# Gather facts about a host
ansible localhost -m setup

Playbooks

Playbooks are YAML files that define automation workflows. Here is a playbook to set up a web server:

---
- name: Configure web server
 hosts: webservers
 become: yes
 vars:
  http_port: 80
  doc_root: /var/www/myapp

 tasks:
  - name: Install Nginx
   apt:
    name: nginx
    state: present

  - name: Create document root
   file:
    path: "{{ doc_root }}"
    state: directory
    owner: www-data
    group: www-data

  - name: Copy site config
   template:
    src: nginx.conf.j2
    dest: /etc/nginx/sites-available/myapp

  - name: Enable site
   file:
    src: /etc/nginx/sites-available/myapp
    dest: /etc/nginx/sites-enabled/myapp
    state: link

  - name: Ensure Nginx is running
   systemd:
    name: nginx
    state: started
    enabled: yes

Run a playbook with:

ansible-playbook -i inventory setup-webserver.yml

Variables & Templates

Variables keep playbooks reusable. Define them in group_vars/, host_vars/, or inline:

# group_vars/webservers.yml
---
nginx_port: 8080
app_repo: "https://github.com/org/myapp.git"
log_level: info

Use templates (.j2 files) to generate configs dynamically:

# templates/nginx.conf.j2
server {
  listen {{ nginx_port }};
  server_name {{ ansible_fqdn }};
  root /var/www/{{ app_name }};
  access_log /var/log/nginx/{{ app_name }}.log;
}

Jinja2 filters like {{ variable | default('fallback') }} handle missing values gracefully.

Roles

Roles organize playbooks into reusable components. Structure:

roles/
 nginx/
  tasks/main.yml
  handlers/main.yml
  templates/
  files/
  vars/main.yml
  defaults/main.yml
  meta/main.yml

Use roles in a playbook:

---
- name: Apply common config
 hosts: all
 roles:
  - common
  - monitoring

- name: Web server specific
 hosts: webservers
 roles:
  - nginx
  - nodejs

Best Practices

  • Use ansible-lint to validate playbooks for common mistakes and anti-patterns.
  • Pin versions — specify package versions in your playbooks to avoid unexpected upgrades.
  • Test with --check — run ansible-playbook --check --diff to see what will change without making changes.
  • Use tags — add tags: [nginx, config] to tasks so you can run only specific parts: --tags nginx.
  • Keep secrets in Vault — never put passwords, API keys, or SSH keys in plaintext in your playbooks.
  • Version control everything — store inventory, playbooks, and roles in Git.

Ansible Vault

Vault encrypts sensitive data so you can store it safely in version control:

# Create an encrypted file
ansible-vault create secrets.yml

# Edit an existing vault
ansible-vault edit secrets.yml

# Encrypt a plaintext file
ansible-vault encrypt vars/passwords.yml

# Decrypt (use with care)
ansible-vault decrypt secrets.yml

# Run a playbook with vault
ansible-playbook playbook.yml --ask-vault-pass

Vault encrypts individual files with AES-256. The password can also come from a file: --vault-password-file .vault_pass.
