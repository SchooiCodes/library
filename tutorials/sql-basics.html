🗃️ SQL Database Basics

Learn Structured Query Language — from simple SELECT queries to complex JOINs and subqueries.

What is SQL?

SQL (Structured Query Language) is the standard language for managing and querying relational databases. Developed in the 1970s at IBM, SQL lets you create, read, update, and delete data stored in tables with defined schemas.

A relational database organizes data into tables (like spreadsheets) where rows represent records and columns represent attributes. Tables can be linked through foreign keys, enabling powerful queries across multiple datasets.

SELECT — Retrieving Data

The SELECT statement is the most fundamental SQL operation. It retrieves rows from one or more tables:

-- Select all columns from a table
SELECT * FROM users;

-- Select specific columns
SELECT id, name, email FROM users;

-- Select with a WHERE filter
SELECT * FROM orders WHERE total > 100;

-- Select with ordering
SELECT name, price FROM products ORDER BY price DESC;

-- Limit the number of results
SELECT * FROM logs LIMIT 10;

INSERT, UPDATE, DELETE

These three commands modify data in your tables:

-- INSERT a new row
INSERT INTO users (name, email, age)
VALUES ('Alice', 'alice@example.com', 30);

-- UPDATE existing rows
UPDATE users SET age = 31 WHERE name = 'Alice';

-- DELETE rows
DELETE FROM users WHERE age < 18;

-- Be careful! DELETE without WHERE removes ALL rows
DELETE FROM users;
Warning: Always use a WHERE clause with UPDATE and DELETE unless you intend to affect every row. A missing WHERE can be catastrophic in production.

JOINs

JOINs combine rows from two or more tables based on a related column. Here is a sample scenario with users and orders tables:

-- INNER JOIN: only matching rows in both tables
SELECT users.name, orders.amount
FROM users
INNER JOIN orders ON users.id = orders.user_id;

-- LEFT JOIN: all rows from left table, NULLs where no match on right
SELECT users.name, orders.amount
FROM users
LEFT JOIN orders ON users.id = orders.user_id;

-- RIGHT JOIN: all rows from right table, NULLs where no match on left
SELECT users.name, orders.amount
FROM users
RIGHT JOIN orders ON users.id = orders.user_id;

-- FULL OUTER JOIN: all rows from both tables
SELECT users.name, orders.amount
FROM users
FULL OUTER JOIN orders ON users.id = orders.user_id;
JOIN TypeRows Returned
INNER JOINOnly rows with matches in both tables
LEFT JOINAll left table rows + matched right rows (or NULL)
RIGHT JOINAll right table rows + matched left rows (or NULL)
FULL OUTER JOINAll rows from both tables, matched where possible

WHERE, GROUP BY, HAVING, ORDER BY

These clauses filter and sort your data:

-- WHERE filters rows before grouping
SELECT * FROM orders WHERE status = 'shipped';

-- GROUP BY groups rows with the same value, used with aggregate functions
SELECT customer_id, COUNT(*) AS order_count, SUM(amount) AS total_spent
FROM orders
WHERE status != 'cancelled'
GROUP BY customer_id;

-- HAVING filters groups (like WHERE but for aggregates)
SELECT customer_id, SUM(amount) AS total
FROM orders
GROUP BY customer_id
HAVING SUM(amount) > 500;

-- ORDER BY sorts the result set
SELECT name, price FROM products ORDER BY price ASC;

Common aggregate functions: COUNT(), SUM(), AVG(), MIN(), MAX().

Order of execution: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT. Understanding this order is critical for debugging complex queries.

Subqueries

A subquery is a query nested inside another query. They can appear in SELECT, FROM, or WHERE clauses:

-- Subquery in WHERE
SELECT name FROM products
WHERE id IN (SELECT product_id FROM order_items WHERE quantity > 5);

-- Subquery in SELECT (scalar subquery)
SELECT name,
  (SELECT AVG(price) FROM products) AS avg_price
FROM products;

-- Subquery in FROM (derived table)
SELECT dept, avg_salary
FROM (SELECT department_id AS dept, AVG(salary) AS avg_salary FROM employees GROUP BY department_id) AS sub
WHERE avg_salary > 70000;

Indexes

Indexes speed up data retrieval at the cost of slower writes and additional storage:

-- Create an index on a single column
CREATE INDEX idx_users_email ON users(email);

-- Create a unique index (enforces uniqueness)
CREATE UNIQUE INDEX idx_users_username ON users(username);

-- Composite index (multiple columns)
CREATE INDEX idx_orders_user_status ON orders(user_id, status);

-- Drop an index
DROP INDEX idx_users_email;

Use EXPLAIN ANALYZE (PostgreSQL) or EXPLAIN (MySQL) to see if your query uses indexes effectively. Index columns used in WHERE, JOIN, and ORDER BY clauses.
