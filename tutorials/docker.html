What is Docker?

Docker is a platform for developing, shipping, and running applications inside lightweight, isolated environments called containers. Unlike virtual machines, containers share the host operating system kernel, which makes them far more efficient in terms of memory, startup time, and disk usage.

Docker was first released in 2013 and has since become the industry standard for containerization. It solves the classic "it works on my machine" problem by packaging an application with all its dependencies into a single portable image.

Images vs Containers

These two terms are often confused but are distinct concepts:

ConceptAnalogyDescription
ImageClass / BlueprintA read-only template with instructions for creating a container. Built from a Dockerfile.
ContainerInstance / HouseA runnable instance of an image. You can start, stop, move, or delete a container.

Think of an image as a class in OOP and a container as an object instantiated from that class. You can have many containers from the same image, each running independently.

Dockerfile Basics

A Dockerfile is a text file that defines how to build a Docker image. Here is a typical example for a Node.js app:

# Use an official Node.js runtime as base
FROM node:20-alpine

# Set working directory
WORKDIR /app

# Copy package files and install dependencies
COPY package*.json ./
RUN npm install

# Copy the rest of the application code
COPY . .

# Expose the port the app runs on
EXPOSE 3000

# Command to run the application
CMD ["node", "server.js"]

Key instructions:

  • FROM — Sets the base image (always the first instruction)
  • WORKDIR — Sets the working directory for subsequent instructions
  • COPY / ADD — Copies files from host to image
  • RUN — Executes commands during image build
  • EXPOSE — Documents which port the container listens on
  • CMD / ENTRYPOINT — Defines the default command when the container starts
  • ENV — Sets environment variables
Best Practice: Use multi-stage builds to keep images small. One stage builds your app with all build tools, then a second slim stage copies only the compiled artifacts.

docker-compose.yml

Docker Compose lets you define and run multi-container applications with a single YAML file. Here is an example with a web app and a PostgreSQL database:

version: '3.9'

services:
 web:
  build: .
  ports:
   - "3000:3000"
  environment:
   - DATABASE_URL=postgres://user:pass@db:5432/mydb
  depends_on:
   - db
  volumes:
   - .:/app
   - /app/node_modules

 db:
  image: postgres:16-alpine
  environment:
   POSTGRES_USER: user
   POSTGRES_PASSWORD: pass
   POSTGRES_DB: mydb
  volumes:
   - pgdata:/var/lib/postgresql/data

volumes:
 pgdata:

Run with docker compose up -d and tear down with docker compose down. Compose handles networking between services automatically — services can reach each other by their service name.

Common Commands

CommandDescription
docker run -d -p 8080:80 nginxRun an nginx container in detached mode, mapping host 8080 to container 80
docker build -t myapp:latest .Build an image from the Dockerfile in the current directory
docker psList running containers
docker ps -aList all containers (including stopped)
docker exec -it <container> bashOpen an interactive shell inside a running container
docker logs -f <container>Follow log output of a container
docker compose up -dStart services defined in compose.yml in detached mode
docker compose downStop and remove all containers, networks, and volumes defined in compose.yml
docker imagesList all images on the host
docker rmi <image>Remove an image
docker rm <container>Remove a stopped container
docker pull <image>Pull an image from a registry
docker push <image>Push an image to a registry
docker system pruneClean up unused data (containers, networks, images, build cache)

Best Practices

  • Use specific tags: Avoid latest in production. Pin to node:20-alpine, not node:latest.
  • Minimize layers: Combine RUN commands with && to reduce image size.
  • Use .dockerignore: Exclude node_modules, .git, and other unnecessary files from the build context.
  • Run as non-root: Use the USER directive to avoid running containers as root.
  • Health checks: Add HEALTHCHECK to your Dockerfile so orchestrators know if your app is alive.
  • One process per container: Don't run multiple services (e.g., Node + PostgreSQL) in the same container.

Volumes & Networks

Volumes persist data generated by containers. Unlike bind mounts (which map a host directory directly), volumes are fully managed by Docker and are the recommended way to persist data:

# Create a named volume
docker volume create mydata

# Use a volume in a container
docker run -v mydata:/data myimage

Networks allow containers to communicate. Docker provides several network drivers:

  • bridge — Default. Containers on the same bridge can talk to each other.
  • host — Removes network isolation, containers share the host's network stack.
  • none — No networking.
  • overlay — Connects multiple Docker daemons across machines (used in Swarm).

Docker Hub & Registries

Docker Hub is the default public registry where you can find thousands of official images (nginx, node, postgres, python, ubuntu). You can also push your own images:

# Tag an image for a registry
docker tag myapp username/myapp:latest

# Log in to Docker Hub
docker login

# Push the image
docker push username/myapp:latest

Other popular registries include GitHub Container Registry (ghcr.io), GitLab Container Registry, Amazon ECR, Google Artifact Registry, and Quay.io. Private registries are essential for proprietary code.

Docker vs Podman vs Traditional VMs

FeatureDockerPodmanTraditional VM
DaemonRequires a running daemon (dockerd)Daemonless — runs containers as child processesRequires hypervisor (Hyper-V, VirtualBox, VMware)
RootlessSupported (since v19.03)Native rootless (no setuid binary needed)Depends on hypervisor
OrchestrationSwarm, KubernetesPodman pods (Kubernetes-compatible YAML)Via hypervisor API
Image formatOCI-compliantOCI-compliant (same images as Docker)VMDK, VHDX, QCOW2
Startup timeMillisecondsMillisecondsSeconds to minutes
Memory overheadMinimal (MBs)Minimal (MBs)GBs (full OS per VM)
IsolationKernel namespaces + cgroupsKernel namespaces + cgroupsFull hardware-level virtualization
CLI compatibilitydocker CLIpodman CLI (alias docker → podman)Hypervisor-specific

Podman is a daemonless, rootless alternative to Docker with identical CLI syntax — you can often alias alias docker=podman. VMs provide stronger isolation (different kernel) but at a much higher resource cost. For most development and microservice deployments, containers are the right choice.

Getting Started: Install Docker from docs.docker.com/get-docker, then run docker run hello-world to verify your installation.
