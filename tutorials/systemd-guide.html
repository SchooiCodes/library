⚙️ Systemd Service Guide

Create and manage systemd services, timers, sockets, and targets — journalctl, unit files, and troubleshooting.

What is systemd?

systemd is the init system and service manager used by most modern Linux distros. It boots the system, manages daemons, handles logging, and controls system state. Units (services, timers, sockets, targets) are defined by unit files.

# Check your systemd version
systemd --version

# List all active units
systemctl list-units

# List all installed unit files
systemctl list-unit-files

Systemctl Basics

systemctl is the main command to interact with systemd. Control services, check status, enable/disable startup behavior.

# Service lifecycle
sudo systemctl start nginx
sudo systemctl stop nginx
sudo systemctl restart nginx
sudo systemctl reload nginx

# Status
systemctl status nginx
systemctl is-active nginx
systemctl is-enabled nginx

# Enable/disable at boot
sudo systemctl enable nginx
sudo systemctl disable nginx
sudo systemctl enable --now nginx  # enable + start

# Daemon reload (after editing unit files)
sudo systemctl daemon-reload

Writing Unit Files

Unit files live in /etc/systemd/system/ (system) or /lib/systemd/system/ (distro-provided). A service unit has three main sections: [Unit], [Service], and [Install].

# /etc/systemd/system/myapp.service
[Unit]
Description=My Custom Application
After=network.target
Wants=network-online.target

[Service]
Type=simple
User=myapp
Group=myapp
WorkingDirectory=/opt/myapp
ExecStart=/usr/bin/python3 /opt/myapp/app.py
ExecStop=/bin/kill $MAINPID
Restart=on-failure
RestartSec=5
Environment="APP_ENV=production"

[Install]
WantedBy=multi-user.target
Tip: After creating or editing a unit file, run sudo systemctl daemon-reload to reload the configuration.

Journalctl Logging

systemd's journal captures logs from all services and the kernel. journalctl queries this binary log with powerful filtering options.

# View all logs
journalctl

# Follow new log entries (like tail -f)
journalctl -f

# Logs for a specific service
journalctl -u nginx

# Since last boot
journalctl -b

# Since a specific time
journalctl --since "1 hour ago"
journalctl --since "2026-05-01" --until "2026-05-02"

# Kernel messages only
journalctl -k

# Export JSON
journalctl -u nginx -o json

Timers (cron replacement)

systemd timers replace cron for scheduled tasks. A timer unit triggers a service unit on a calendar schedule or monotonic interval. Timers are more reliable than cron and have built-in logging.

# /etc/systemd/system/backup.service
[Unit]
Description=Daily backup

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh

# /etc/systemd/system/backup.timer
[Unit]
Description=Run backup daily at 2 AM

[Timer]
OnCalendar=daily
Persistent=true

[Install]
WantedBy=timers.target
sudo systemctl enable --now backup.timer
systemctl list-timers

Targets (runlevels)

Targets group units into system states. They replace the old SysV runlevels. Common targets: poweroff.target, rescue.target, multi-user.target (text mode), graphical.target (with GUI).

# View default target
systemctl get-default

# Set default target
sudo systemctl set-default multi-user.target

# Switch target now
sudo systemctl isolate multi-user.target

# List all targets
systemctl list-units --type=target

Debugging Services

When a service fails, check its status and journal. Common issues include incorrect paths, permission problems, and missing environment variables.

# Check service status
systemctl status myapp.service

# Show recent journal entries for the failed service
journalctl -u myapp.service -n 50 --no-pager

# Follow logs while restarting
journalctl -u myapp.service -f &
sudo systemctl restart myapp

# Test the ExecStart command manually as the service user
sudo -u myapp /usr/bin/python3 /opt/myapp/app.py

# Verify unit file syntax
systemd-analyze verify /etc/systemd/system/myapp.service
Tip: Use SYSTEMD_LOG_LEVEL=debug for verbose debug output: env SYSTEMD_LOG_LEVEL=debug systemctl status myapp.
