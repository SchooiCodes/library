⚙️ Task Automation Guide

Automate everything — cron, systemd timers, scripts, and CI/CD pipelines.

Why Automate?

Task automation eliminates repetitive manual work, reduces human error, and ensures consistency. From simple file backups to complex CI/CD pipelines, automation is a core skill for any system administrator or developer. This guide covers the major scheduling tools and scripting approaches.

Linux Cron Jobs

Cron is the classic Unix job scheduler. It runs commands at specified times using crontab files:

# Edit your crontab
crontab -e

# Crontab syntax:
# * * * * * command_to_execute
# | | | | |
# | | | | +---- Day of week (0-7, 0=Sunday)
# | | | +------ Month (1-12)
# | | +-------- Day of month (1-31)
# | +---------- Hour (0-23)
# +------------ Minute (0-59)

# Examples:
# Run every day at 2:30 AM
30 2 * * * /home/user/scripts/backup.sh

# Run every hour
0 * * * * /usr/bin/python3 /home/user/scripts/cleanup.py

# Run every Monday at 9 AM
0 9 * * 1 /home/user/scripts/weekly-report.sh

# Run every 15 minutes
*/15 * * * * /home/user/scripts/check-status.sh

# Run on the 1st of every month at midnight
0 0 1 * * /home/user/scripts/monthly-maintenance.sh

Special strings for common schedules:

StringMeaning
@rebootRun once after system boots
@hourlyRun at the start of every hour (0 * * * *)
@dailyRun at midnight (0 0 * * *)
@weeklyRun at midnight on Sunday (0 0 * * 0)
@monthlyRun at midnight on the 1st (0 0 1 * *)
Note: Cron jobs run with minimal environment — no PATH, no GUI. Always use full paths to executables and redirect output: 0 2 * * * /usr/bin/python3 /home/user/script.py >> /var/log/mylog.log 2>&1.

systemd Timers

systemd timers are the modern alternative to cron on Linux. They offer more features like calendar events, monotonic timers (relative to boot), and dependency management:

# /etc/systemd/system/backup.service
[Unit]
Description=Daily database backup

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup-db.sh
User=postgres
# /etc/systemd/system/backup.timer
[Unit]
Description=Run backup daily at 3 AM

[Timer]
OnCalendar=daily
OnCalendar=*-*-* 03:00:00
Persistent=true    # Catch up after missed runs (e.g., after boot)

[Install]
WantedBy=timers.target
# Enable and start the timer
sudo systemctl enable backup.timer
sudo systemctl start backup.timer

# List active timers
systemctl list-timers --all

systemd timers are better than cron when your task has dependencies on other systemd units (e.g., run after postgresql.service is up).

Windows Task Scheduler

Windows has its own built-in scheduler accessible via GUI or PowerShell:

# Create a scheduled task via PowerShell
$action = New-ScheduledTaskAction -Execute "powershell.exe" -Argument "-File C:\Scripts\backup.ps1"
$trigger = New-ScheduledTaskTrigger -Daily -At "03:00AM"
$principal = New-ScheduledTaskPrincipal -UserId "SYSTEM" -LogonType ServiceAccount
Register-ScheduledTask -TaskName "DailyBackup" -Action $action -Trigger $trigger -Principal $principal

Use taskschd.msc to open the GUI Task Scheduler for manual configuration. You can set triggers (time, event, login), actions (start program, send email, show message), and conditions (only run on AC power, idle only, network available).

Bash / Python Scripting for Automation

Most automation tasks boil down to a script that cron or a timer invokes. Here are practical examples:

Bash Backup Script

#!/bin/bash
# /usr/local/bin/backup-db.sh
BACKUP_DIR="/var/backups/postgres"
DB_NAME="mydb"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
RETENTION_DAYS=30

mkdir -p "$BACKUP_DIR"
pg_dump "$DB_NAME" | gzip > "$BACKUP_DIR/${DB_NAME}_${TIMESTAMP}.sql.gz"
find "$BACKUP_DIR" -name "*.sql.gz" -mtime +$RETENTION_DAYS -delete
echo "Backup completed: ${DB_NAME}_${TIMESTAMP}.sql.gz"

Python Cleanup Script

#!/usr/bin/env python3
# /usr/local/bin/cleanup-temp.py
import os, time, shutil

TEMP_DIRS = ['/tmp', '/var/tmp']
MAX_AGE_HOURS = 24

now = time.time()
for d in TEMP_DIRS:
    for root, dirs, files in os.walk(d):
        for name in files + dirs:
            path = os.path.join(root, name)
            try:
                age_hours = (now - os.path.getmtime(path)) / 3600
                if age_hours > MAX_AGE_HOURS:
                    if os.path.isfile(path):
                        os.remove(path)
                    else:
                        shutil.rmtree(path)
            except (OSError, PermissionError):
                pass

CI/CD Concepts — GitHub Actions Basics

CI/CD (Continuous Integration / Continuous Deployment) automates the build, test, and deployment pipeline. GitHub Actions is one of the most popular CI/CD platforms:

# .github/workflows/deploy.yml
name: Deploy to Production

on:
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm test

  deploy:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /opt/myapp
            git pull
            npm ci
            npm run build
            sudo systemctl restart myapp

Key concepts: workflows (YAML files), jobs (sequential or parallel tasks), steps (individual commands or actions), secrets (encrypted environment variables), and runners (the machines that execute jobs).

Real-World Examples

  • Daily database backup — cron + pg_dump + gzip + S3 upload (rclone)
  • Log rotation — logrotate (pre-installed on most distros), not cron, but closely related
  • SSL certificate renewal — certbot with --cron or systemd timer
  • System updates — unattended-upgrades for security patches
  • Notification scripts — curl to Discord/Slack webhooks when a service goes down
  • Disk cleanup — find + rm for old logs, Docker system prune, tmpwatch/tmpreaper
  • Health check — curl a URL every 5 minutes, alert if non-200 response
Golden rule of automation: Always add logging. Every automated task should write to a log file, and you should monitor those logs. A silent failure is worse than no automation at all.
