Microservices Architecture

Microservices -- design, communication, data management, and deployment.

Table of Contents (2 sections)

  • Design & Communication
  • Data & Deployment

Design & Communication

Microservices: single responsibility, independently deployable. Bounded context: domain-driven design (DDD). Service boundaries: align to business capabilities (orders, payments, inventory). Inter-service communication: Synchronous (REST, gRPC -- request/reply). gRPC: Protocol Buffers, HTTP/2, streaming -- stub/skeleton generation. Asynchronous: message queues (Kafka, RabbitMQ), event-driven. API Gateway: single entry point -- auth, routing, rate limiting, aggregation (BFF pattern -- Backend For Frontend). Service discovery: DNS, Consul, Kubernetes DNS. Resiliency: circuit breaker (Resilience4j, Hystrix), retry, timeout, bulkhead. Observability: distributed tracing (Jaeger, Zipkin -- trace ID propagation). Health checks: /health, /ready liveness probes.

Data & Deployment

Database per service: each service owns its data (no shared DB). Eventual consistency: Saga pattern (choreography -- events, orchestration -- coordinator). CQRS: separate read/write models. Event sourcing: store events, derive current state. API composition: aggregate data from multiple services. Strangler Fig pattern: gradually migrate monolith to microservices. Deployment: containers (Docker), orchestration (K8s, Nomad). CI/CD: each service has its own pipeline. Canary deployments: gradual rollout. Blue/green: switch between environments. Service mesh: Istio, Linkerd -- traffic management, mTLS, observability, circuit breaker. Challenges: network latency, distributed transactions, testing complexity, debugging, data consistency. Monolith-first: start with monolith, extract microservices when needed. When NOT to use: small team, simple domain, early stage, no scaling need.
