🔄 rsync

Fast, versatile file synchronization and remote backup tool for Unix-like systems.

What is rsync?

rsync is a fast, versatile file copying and synchronization tool for Unix-like systems. It efficiently transfers and syncs files between directories on the same machine or across a network using delta encoding.

The key innovation of rsync is its delta-transfer algorithm — it only transfers the differences between source and destination files, making repeated syncs extremely fast. It also supports compression, exclusion patterns, partial transfers, and preserving file permissions, ownerships, and timestamps.

Written by Andrew Tridgell and Paul Mackerras in 1996, rsync is the standard tool for backups, mirroring, and large file transfers on Linux and macOS.

Basic Syntax

The basic rsync command syntax is:

rsync [options] source destination

# Source can be a file, directory, or remote path
# Destination can be a local path or remote path

# Simple local file copy
rsync file.txt backup/file.txt

# Copy a directory (trailing slash matters!)
rsync -a /source/dir/ /destination/dir/

# Copy directory contents (with trailing slash on source)
rsync -a /source/dir/ /destination/dir/
# Creates /destination/dir/ containing source files

# Copy directory itself (no trailing slash)
rsync -a /source/dir /destination/
# Creates /destination/dir/ containing source files

The trailing slash after the source path is critical: with a slash, it copies the contents of the directory; without, it copies the directory itself.

Tip: Always use -a (archive mode) for backup operations. It preserves permissions, ownership, timestamps, and recursively copies directories.

Local & Remote Sync

rsync works seamlessly across local and remote locations:

# Local to local
rsync -av /home/user/documents/ /mnt/backup/documents/

# Local to remote (push)
rsync -av /home/user/projects/ user@server:/backup/projects/

# Remote to local (pull)
rsync -av user@server:/backup/projects/ /home/user/restored/

# Remote to remote (direct)
rsync -av user1@server1:/data/ user2@server2:/data/

# Using different SSH port
rsync -av -e "ssh -p 2222" /local/ user@server:/remote/

For remote transfers, rsync uses SSH by default. The colon (:) separates the host from the path. Single colon uses SSH; double colon (::) uses the rsync daemon protocol.

Common Options Explained

rsync has a vast number of options. Here are the most important ones:

OptionDescription
-a, --archiveRecursive with preserved permissions, timestamps, owner, group
-v, --verboseShow detailed output of what's being transferred
-z, --compressCompress data during transfer (good for slow connections)
-h, --human-readableShow numbers in human-readable format
--progressShow progress for each file
--deleteRemove files at destination not present at source
--exclude=PATTERNExclude files matching pattern
--include=PATTERNInclude files matching pattern (overrides exclude)
--dry-runShow what would happen without actually transferring
-nSame as --dry-run
--partialKeep partially transferred files (useful for large files)
--bwlimit=RATELimit I/O bandwidth (KB/s)
-PSame as --partial --progress

SSH & Remote Sync

rsync over SSH is the standard for remote transfers. Configure SSH keys for passwordless automated backups:

# Generate SSH key
ssh-keygen -t ed25519 -C "backup@example.com"

# Copy key to remote server
ssh-copy-id user@backup-server.com

# Now rsync without password prompts
rsync -avz /data/ user@backup-server.com:/backups/data/

# Use custom SSH options
rsync -avz -e "ssh -i ~/.ssh/backup_key -p 2222" \
  /data/ user@backup-server.com:/backups/

# rsync over SSH with compression
rsync -avz --delete -e "ssh -o StrictHostKeyChecking=no" \
  /source/ user@server:/dest/

# Limit bandwidth for remote sync (500 KB/s)
rsync -avz --bwlimit=500 /data/ user@server:/backups/

Using SSH keys is essential for cron-based automated backups. Consider using an SSH agent or ssh-agent for passphrase-protected keys.

Incremental Backups

rsync is perfect for incremental backups — after the initial full copy, subsequent runs only transfer changed data:

# First full backup
rsync -av /home/user/data/ /backup/data/

# Subsequent incremental backups (same command)
# Only changed files are transferred!
rsync -av /home/user/data/ /backup/data/

# Backup with hard links for version history
# Use rsync with --link-dest for previous backup as reference
rsync -av --delete \
  --link-dest=/backup/current \
  /source/ \
  /backup/$(date +%Y-%m-%d)/
  
# Create/update "current" symlink
ln -snf /backup/$(date +%Y-%m-%d) /backup/current

The --link-dest technique creates hard links to unchanged files from the previous backup, saving enormous amounts of disk space while keeping each snapshot as a complete directory tree.

Exclude Patterns

Exclude files and directories from sync using patterns:

# Exclude a directory
rsync -av --exclude='node_modules' /project/ /backup/project/

# Exclude multiple patterns
rsync -av \
  --exclude='.git' \
  --exclude='*.log' \
  --exclude='.cache' \
  --exclude='__pycache__' \
  --exclude='tmp/' \
  /project/ /backup/project/

# Include specific files while excluding everything else
rsync -av \
  --include='*.html' \
  --include='*.css' \
  --include='*.js' \
  --exclude='*' \
  /project/ /backup/project/

# Exclude file list
# In exclude-patterns.txt:
# *.log
# .git/
# tmp/
# node_modules/

rsync -av --exclude-from=exclude-patterns.txt /src/ /dest/

Dry Run Mode

Always use --dry-run (or -n) to preview changes before executing:

# Preview what would be transferred
rsync -av --dry-run /source/ /destination/

# Preview with delete (see what would be removed)
rsync -av --dry-run --delete /source/ /destination/

# Preview with output summary
rsync -av --dry-run --stats /source/ /destination/

# Compare dry run output with actual run
# Dry run shows:
# sending incremental file list
# file1.txt
# file2.txt
# 
# sent 100 bytes received 20 bytes 240.00 bytes/sec
# total size is 50000 speedup is 416.67 (DRY RUN)

# Remove --dry-run to actually execute

Dry run is especially critical when using --delete — it shows you which files will be removed before they're actually gone.

Performance Tuning

Optimize rsync performance for different scenarios:

# For many small files: use a larger checksum seed
rsync -av --checksum-seed=32768 /src/ /dest/

# For slow connections: compress and limit bandwidth
rsync -avz --bwlimit=1000 /src/ user@remote:/dest/

# For fast LAN: skip compression (it wastes CPU)
rsync -av --no-compress /src/ /dest/

# For very large files: use --partial to resume
rsync -av --partial --append /hugefile.iso user@remote:/dest/

# For many files: increase I/O priority
rsync -av --rsync-path="ionice -c 3 rsync" /src/ /dest/

# For SSH: use faster cipher
rsync -avz -e "ssh -c aes128-gcm@openssh.com" /src/ user@remote:/dest/

# Use -W (whole file) only for local transfers or if delta algorithm is slow
rsync -avW /src/ /dest/

Backup Script Example

Here is a complete backup script using rsync with daily, weekly, and monthly rotation:

#!/bin/bash
# backup.sh — Automated rsync backup with rotation

SOURCE="/home/user/data"
DEST_BASE="/mnt/backup"
DATE=$(date +%Y-%m-%d)
LOG="/var/log/backup.log"

# Determine backup type based on day of week
DAY_OF_WEEK=$(date +%u)
DAY_OF_MONTH=$(date +%d)

if [ "$DAY_OF_MONTH" = "01" ]; then
  LABEL="monthly/$DATE"
elif [ "$DAY_OF_WEEK" = "7" ]; then
  LABEL="weekly/$DATE"
else
  LABEL="daily/$DATE"
fi

DEST_DIR="$DEST_BASE/$LABEL"
LATEST="$DEST_BASE/daily/latest"

# Create destination
mkdir -p "$DEST_DIR"

# Run rsync with hard links to previous backup
rsync -av --delete \
  --link-dest="$LATEST" \
  --exclude='.cache' \
  --exclude='node_modules' \
  --exclude='*.tmp' \
  --log-file="$LOG" \
  "$SOURCE/" "$DEST_DIR/"

# Update latest symlink
rm -f "$LATEST"
ln -s "$DEST_DIR" "$LATEST"

# Cleanup: keep 7 daily, 4 weekly, 3 monthly
find "$DEST_BASE/daily" -maxdepth 1 -type d ! -name "latest" -mtime +7 -exec rm -rf {} +
find "$DEST_BASE/weekly" -maxdepth 1 -type d -mtime +28 -exec rm -rf {} +
find "$DEST_BASE/monthly" -maxdepth 1 -type d -mtime +90 -exec rm -rf {} +

echo "Backup complete: $DEST_DIR" | tee -a "$LOG"
