Functions & Parameters

Functions are the building blocks of reusable PowerShell code. Define them with function, use param() for typed parameters, and return data with return or simply by outputting objects:

function Get-SystemInfo {
  param(
    [Parameter(Mandatory=$false)]
    [string[]]$ComputerName = @($env:COMPUTERNAME),

    [ValidateSet('CPU','RAM','Disk','OS')]
    [string[]]$Category = @('CPU','RAM','Disk','OS')
  )

  foreach ($comp in $ComputerName) {
    if ($Category -contains 'OS') {
      Get-CimInstance -ComputerName $comp Win32_OperatingSystem |
        Select-Object PSComputerName, Caption, Version, BuildNumber
    }
    if ($Category -contains 'CPU') {
      Get-CimInstance -ComputerName $comp Win32_Processor |
        Select-Object PSComputerName, Name, NumberOfCores, MaxClockSpeed
    }
  }
}

Get-SystemInfo -ComputerName 'PC1','PC2' -Category OS,CPU

Modules & Scripts

A module is a package of related functions, classes, and variables. Save as .psm1 with a matching manifest .psd1. Organise shared tools into modules for easy reuse:

# Create a simple module
# Save as MyToolbox.psm1
function Invoke-HealthCheck {
  Write-Host "Checking system health..." -ForegroundColor Green
  Get-CimInstance Win32_LogicalDisk -Filter "DriveType=3" |
    Select-Object DeviceID, @{N='FreeGB';E={[math]::Round($_.FreeSpace/1GB,2)}}
}

# Create module manifest
New-ModuleManifest -Path .\MyToolbox.psd1 -ModuleVersion "1.0" -RootModule MyToolbox.psm1

# Import and use
Import-Module .\MyToolbox.psd1
Invoke-HealthCheck

# Publish functions via script file (.ps1)
# . .\my-script.ps1 — dot-source to load into current session

Error Handling (try/catch)

PowerShell supports structured exception handling. Use try/catch/finally and set $ErrorActionPreference to control behaviour:

function Safe-RemoveDirectory {
  param([string]$Path)

  $ErrorActionPreference = 'Stop'
  try {
    if (Test-Path $Path) {
      Remove-Item $Path -Recurse -Force
      Write-Host "Removed: $Path" -ForegroundColor Green
    } else {
      Write-Warning "Path not found: $Path"
    }
  }
  catch [System.UnauthorizedAccessException] {
    Write-Error "Access denied: $Path. Run as Administrator."
  }
  catch [System.IO.PathTooLongException] {
    Write-Error "Path too long: $Path"
  }
  catch {
    Write-Error "Unexpected error: $_"
  }
  finally {
    $ErrorActionPreference = 'Continue'
  }
}

# $Error is an automatic array of recent errors
$Error[0] | Format-List * -Force

Remoting (WinRM, Enter-PSSession)

PowerShell Remoting uses WinRM to execute commands on remote machines. Enable it with Enable-PSRemoting and connect with Enter-PSSession or Invoke-Command:

# Enable remoting on the target machine (Admin prompt)
Enable-PSRemoting -Force

# Trust all hosts on the local subnet (or specify IPs)
Set-Item WSMan:\localhost\Client\TrustedHosts -Value "192.168.1.*" -Force

# Interactive session
Enter-PSSession -ComputerName "PC01" -Credential (Get-Credential)

# One-off command execution
Invoke-Command -ComputerName "PC01","PC02" -ScriptBlock {
  Get-Service -Name "wuauserv","BITS" | Select-Object Status, DisplayName
} -Credential (Get-Credential)

# Using sessions for efficiency
$session = New-PSSession -ComputerName "PC01","PC02"
Invoke-Command -Session $session -ScriptBlock { Get-Process | Sort-Object CPU -Desc | Select -First 5 }
Remove-PSSession $session
Tip: Run Test-WSMan -ComputerName "target" to verify WinRM is listening. If it fails, check the firewall — WinRM uses TCP 5985 (HTTP) and 5986 (HTTPS). Group Policy can also block remoting.

Scheduled Jobs

PowerShell Scheduled Jobs run scripts on a trigger without needing Task Scheduler. They persist across reboots and run as background jobs:

# Register a scheduled job
$trigger = New-JobTrigger -Daily -At "03:00AM"
$options = New-ScheduledJobOption -WakeToRun -StartIfIdle
Register-ScheduledJob -Name "DailyCleanup" `
  -ScriptBlock {
    $limit = (Get-Date).AddDays(-30)
    Get-ChildItem "C:\Logs" -Recurse | Where-Object { $_.LastWriteTime -lt $limit } | Remove-Item
  } `
  -Trigger $trigger -ScheduledJobOption $options

# List scheduled jobs
Get-ScheduledJob

# Run a scheduled job immediately
Start-Job -DefinitionName "DailyCleanup"
Get-Job -Name "DailyCleanup" | Receive-Job

Execution Policy

PowerShell's execution policy controls which scripts can run. It is not a security boundary (it can be bypassed), but it prevents accidental execution:

# View current policy
Get-ExecutionPolicy

# Set policy (per-user)
Set-ExecutionPolicy RemoteSigned -Scope CurrentUser

# Bypass for a single script (run this way)
powershell.exe -ExecutionPolicy Bypass -File ".\script.ps1"

# Unrestricted (not recommended for production)
Set-ExecutionPolicy Unrestricted -Scope LocalMachine
Danger Zone: Never set ExecutionPolicy to Unrestricted on a production machine. Always use RemoteSigned (allow local scripts, require remote scripts to be signed). Scripts downloaded from the internet may be blocked — right-click the file → Properties → Unblock, or use Unblock-File.

Practical Examples

📁 Bulk File Renamer

Get-ChildItem "C:\Photos\*.jpg" | ForEach-Object {
  $newName = "{0:yyyy-MM-dd}_{1}" -f $_.CreationTime, $_.Name
  Rename-Item $_.FullName -NewName $newName
}

🔍 Find Large Files

Get-ChildItem "C:\" -Recurse -ErrorAction SilentlyContinue |
  Where-Object { $_.Length -gt 1GB } |
  Select-Object FullName, @{N='SizeGB';E={[math]::Round($_.Length/1GB,2)}} |
  Sort-Object SizeGB -Descending

📊 Export All Installed Software

Get-CimInstance Win32_Product |
  Select-Object Name, Version, Vendor, InstallDate |
  Export-Csv "C:\Reports\installed-software.csv" -NoTypeInformation

🔄 Rotate IIS Log Files

$cutoff = (Get-Date).AddDays(-7)
Get-ChildItem "C:\inetpub\logs\LogFiles" -Filter "*.log" -Recurse |
  Where-Object { $_.LastWriteTime -lt $cutoff } |
  Compress-Archive -DestinationPath "C:\LogArchive\iis-logs.zip" -CompressionLevel Optimal
Get-ChildItem "C:\inetpub\logs\LogFiles" -Filter "*.log" -Recurse |
  Where-Object { $_.LastWriteTime -lt $cutoff } | Remove-Item
