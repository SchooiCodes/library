Cloud Backup Strategies

Comprehensive guide to backing up your data using cloud providers, encryption, and automation.

Overview

Cloud backup is the practice of sending copies of your important data to remote servers operated by cloud storage providers. A good backup strategy ensures that data loss from hardware failure, ransomware, theft, fire, or natural disaster is merely an inconvenience rather than a catastrophe.

This guide covers the full spectrum: selecting providers, encrypting data before it leaves your machine (zero-trust model), automating backups with cron and systemd timers, and regularly testing restores so you're never caught off guard.

3-2-1 Backup Rule

The 3-2-1 rule is the gold standard for data protection:

  • 3 — Keep at least three copies of your data (1 primary + 2 backups).
  • 2 — Use at least two different storage media or providers (e.g. external HDD + cloud).
  • 1 — Keep at least one copy offsite (away from your physical location).

Example implementation:

Copy 1: /home/user/            (primary — laptop SSD)
Copy 2: /mnt/backup_drive/     (local — external USB HDD)
Copy 3: rclone remote:backup/  (offsite — Backblaze B2 or S3)
Critical: RAID is NOT a backup. RAID protects against drive failure but not against accidental deletion, corruption, ransomware, or disasters. Always maintain separate, versioned backups.

Cloud Providers

Here are the most popular cloud storage providers for backup, along with rclone configuration examples for each.

rclone Setup (Prerequisite for All)

# Install rclone
sudo apt install rclone       # Debian/Ubuntu
brew install rclone           # macOS
winget install rclone         # Windows

# Interactive config
rclone config

# Or copy a pre-made config to ~/.config/rclone/rclone.conf

Google Drive

rclone config
> n) New remote
> name: gdrive
> Storage: drive
> client_id: (leave blank for default)
> scope: drive.file
> Follow the OAuth URL in your browser, paste the code.

# Test
rclone ls gdrive:

# Backup command
rclone copy /home/user/Documents gdrive:backup/Documents \
  --progress --verbose --bwlimit 5M

Dropbox

rclone config
> n) New remote
> name: dropbox
> Storage: dropbox
> Follow OAuth flow.

# Backup
rclone sync /home/user/Pictures dropbox:backup/Pictures \
  --progress --backup-dir dropbox:backup/Pictures_old

Amazon S3 (or compatible)

rclone config
> n) New remote
> name: s3
> Storage: s3
> provider: AWS
> env_auth: false
> access_key_id: YOUR_ACCESS_KEY
> secret_access_key: YOUR_SECRET_KEY
> region: us-east-1

# Backup
rclone copy /home/user/Projects s3:my-backup-bucket/projects \
  --progress --s3-chunk-size 64M

Backblaze B2

rclone config
> n) New remote
> name: b2
> Storage: b2
> account: YOUR_APPLICATION_KEY_ID
> key: YOUR_APPLICATION_KEY

# Backup
rclone copy /home/user/important b2:my-bucket/important \
  --progress

Encryption Before Upload

Never upload unencrypted data to a cloud provider you don't control. Use client-side encryption so that even if the provider is compromised, your data remains confidential.

Using rclone crypt

rclone can encrypt files transparently before upload:

# Create an encrypted remote layered on top of your base remote
rclone config
> n) New remote
> name: gdrive-encrypted
> Storage: crypt
> remote: gdrive:backup/encrypted
> password: (generate a strong password)
> password2: (file name encryption — optional but recommended)

# Backup to encrypted remote
rclone copy /home/user/Documents gdrive-encrypted: \
  --progress

Using GPG

For one-off encrypted archives:

# Encrypt a tar archive
tar -czf - ~/Documents | gpg --symmetric --cipher-algo AES256 \
  --passphrase "your-strong-phrase" -o backup-$(date +%F).tar.gz.gpg

# Decrypt
gpg --decrypt backup-2026-05-28.tar.gz.gpg | tar -xzf -

Using age (Modern Alternative)

# Generate key pair
age-keygen -o ~/.config/age/key.txt

# Encrypt
tar -czf - ~/Documents | age -r age1...publickey... > backup.tar.gz.age

# Decrypt
age -d -i ~/.config/age/key.txt backup.tar.gz.age | tar -xzf -

# Encrypt with passphrase
tar -czf - ~/Documents | age -p > backup.tar.gz.age

Using restic (Built-in Encryption)

Restic handles encryption and deduplication automatically:

# Init repo
restic init --repo b2:my-bucket/restic-repo

# Backup
restic --repo b2:my-bucket/restic-repo \
  --password-file ~/.restic-password \
  backup /home/user/Documents

# Restore
restic --repo b2:my-bucket/restic-repo \
  --password-file ~/.restic-password \
  restore latest --target /tmp/restore

Automated Backups

Using Cron (Linux/macOS)

# Edit crontab
crontab -e

# Daily backup at 2 AM
0 2 * * * /home/user/scripts/backup.sh >> /var/log/backup.log 2>&1

Example backup.sh:

#!/bin/bash
set -euo pipefail

DATE=$(date +%F)
LOG="/var/log/backup-$DATE.log"

echo "[$DATE] Starting backup..." >> "$LOG"

# rclone to Backblaze B2
rclone sync /home/user/Documents b2:my-bucket/documents \
  --progress --log-file "$LOG"

# restic encrypted backup
restic --repo b2:my-bucket/restic-repo \
  --password-file ~/.restic-password \
  backup /home/user/Pictures >> "$LOG" 2>&1

echo "[$DATE] Backup complete." >> "$LOG"

Using systemd Timers

# /etc/systemd/system/backup.service
[Unit]
Description=Daily backup service

[Service]
Type=oneshot
ExecStart=/home/user/scripts/backup.sh
User=user

# /etc/systemd/system/backup.timer
[Unit]
Description=Run backup daily

[Timer]
OnCalendar=daily
Persistent=true

[Install]
WantedBy=timers.target
sudo systemctl daemon-reload
sudo systemctl enable --now backup.timer

Using launchd (macOS)

# ~/Library/LaunchAgents/com.user.backup.plist




    Label
    com.user.backup
    ProgramArguments
    
        /home/user/scripts/backup.sh
    
    StartCalendarInterval
    
        Hour
        3
        Minute
        0

Hybrid Strategies

A hybrid approach uses multiple tools and locations for maximum resilience.

Borg + BorgBase

Borg is a deduplicating archiver with compression and encryption. BorgBase offers managed Borg hosting.

# Initialize repo on BorgBase
borg init --encryption=repokey-blake2 \
  ssh://username@repo.borgbase.com:22/./my-repo

# Create backup
borg create \
  --verbose --filter AME \
  --list --stats --show-rc \
  --compression lz4 \
  --exclude-caches \
  --exclude '*.tmp' \
  ssh://username@repo.borgbase.com:22/./my-repo::docs-{now:%Y-%m-%d} \
  /home/user/Documents

# Prune old backups
borg prune \
  --keep-daily 7 --keep-weekly 4 --keep-monthly 6 \
  ssh://username@repo.borgbase.com:22/./my-repo

Restic + rclone

Restic can use rclone as a backend, combining restic's encryption/dedup with rclone's wide provider support:

# Init restic using rclone remote
restic init -r rclone:b2:my-bucket/restic-repo

# Backup
restic -r rclone:b2:my-bucket/restic-repo \
  backup /home/user/Documents

Duplicati

Duplicati provides a web UI for managing backups with AES-256 encryption and deduplication:

# Install
sudo apt install duplicati
# Access web UI at http://localhost:8200

Restore Testing

A backup that has never been tested is not a backup. Schedule regular restore drills:

# Test rclone restore
rclone copy b2:my-bucket/documents /tmp/test-restore \
  --progress

# Test restic restore (latest snapshot)
restic --repo b2:my-bucket/restic-repo \
  --password-file ~/.restic-password \
  restore latest --target /tmp/restic-restore

# List restic snapshots
restic snapshots --repo b2:my-bucket/restic-repo

# Mount a restic snapshot (for selective restore)
restic mount --repo b2:my-bucket/restic-repo /mnt/restic-mount
Recommendation: Every 3 months, restore a random file from each backup location and verify its integrity (open it, check SHA-256, etc.). Once a year, do a full disaster-recovery dry run on a blank machine.

Cost-Saving Tips

  • Use lifecycle policies — Move old backups to cold storage tiers automatically.
  • Deduplicate — Tools like Borg and restic deduplicate across snapshots, saving up to 90% on repeated data.
  • Compress — Enable compression in Borg (lz4 or zstd) or restic. Reduces storage cost by 20-50%.
  • Exclude caches — Don't back up ~/.cache, node_modules/, *.tmp, etc.
  • Monitor bandwidth — Use --bwlimit in rclone to avoid saturating your uplink during work hours.

Troubleshooting

IssueSolution
rclone: "failed to copy"Check network. Verify remote config with rclone config show. Ensure OAuth tokens haven't expired (rclone config reconnect).
restic: "wrong password" on restoreStore your repo password in a password manager. Test password storage during backup setup: echo "mypass" | restic --repo ... snapshots.
Backup takes too longUse --bwlimit during the day, run at night. Enable incremental backups (restic/Borg do this by default). Exclude unnecessary files.
GPG: "cannot decrypt with symmetric algorithm"You used a different GPG version or algorithm. Always note which cipher you used and include it in the filename (e.g. backup-2026-AES256.gpg).
Restore: missing filesCheck that your backup command included the intended paths. Use rclone ls or restic find to verify snapshot contents.
cron job not runningCheck crontab -l. Ensure the script has execute permission. Check journalctl -u cron for errors. Use absolute paths in scripts.
