🌐 REST API Basics

A primer on RESTful APIs — HTTP methods, status codes, authentication, testing, and best practices.

What is an API?

An API (Application Programming Interface) is a set of rules that allows one piece of software to communicate with another. In the context of web development, a REST API exposes endpoints (URLs) that clients call to perform operations on resources, typically returning JSON or XML responses.

REST Principles

REST (Representational State Transfer) is an architectural style defined by Roy Fielding in 2000. RESTful APIs adhere to these principles:

  • Stateless — Each request contains all information needed to process it. No client context is stored on the server between requests.
  • Client-Server — Separation of concerns: the client handles the UI, the server handles data storage and business logic.
  • Cacheable — Responses must explicitly state whether they can be cached to improve performance.
  • Uniform Interface — Consistent URL structure, use of HTTP methods, and standard response formats.
  • Layered System — The client cannot tell whether it is talking directly to the server or to an intermediary (load balancer, cache, etc.).

HTTP Methods

Each HTTP method maps to a CRUD operation on a resource:

MethodCRUDPurposeBodyIdempotent
GETReadRetrieve a resource or collectionNoYes
POSTCreateCreate a new resourceYesNo
PUTUpdate/ReplaceReplace an entire resourceYesYes
PATCHPartial UpdateModify part of a resourceYesNo
DELETEDeleteRemove a resourceMaybeYes
 
# Example API calls to a users resource

GET  /api/users      # List all users
GET  /api/users/42    # Get user with ID 42
POST  /api/users      # Create a new user
PUT  /api/users/42    # Replace user 42 entirely
PATCH /api/users/42    # Update user 42's email only
DELETE /api/users/42    # Delete user 42

Status Codes

HTTP status codes communicate the result of a request. They are grouped into five classes:

CodeNameMeaning
200OKRequest succeeded. Body contains the requested data.
201CreatedA new resource was created (used after POST/PUT).
204No ContentRequest succeeded but no body (used after DELETE).
301/302RedirectThe resource has moved. Follow the Location header.
400Bad RequestThe client sent malformed or invalid data.
401UnauthorizedAuthentication is required or failed.
403ForbiddenThe client is authenticated but not authorized.
404Not FoundThe requested resource does not exist.
409ConflictRequest conflicts with current state (e.g., duplicate).
422Unprocessable EntityValidation failed on the request body.
429Too Many RequestsRate limit exceeded. Retry after the time in Retry-After header.
500Internal Server ErrorSomething went wrong on the server.
502Bad GatewayAn upstream server returned an invalid response.
503Service UnavailableServer is overloaded or under maintenance.

Headers, Request & Response Body

Headers carry metadata about the request or response:

# Request headers
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
Content-Type: application/json
Accept: application/json
User-Agent: MyApp/1.0

# Response headers
Content-Type: application/json; charset=utf-8
Cache-Control: no-cache
RateLimit-Remaining: 99

Request body (for POST, PUT, PATCH) typically contains JSON:

POST /api/users
Content-Type: application/json

{
 "name": "Alice",
 "email": "alice@example.com",
 "role": "admin"
}

Response body returns the requested data or error details:

HTTP/1.1 201 Created
Content-Type: application/json

{
 "id": 42,
 "name": "Alice",
 "email": "alice@example.com",
 "role": "admin",
 "createdAt": "2026-05-27T10:30:00Z"
}

Authentication

APIs need to know who is calling them. Common authentication methods include:

  • API Keys — Simple, static string passed in header or query param. Suitable for server-to-server. Weak security — anyone with the key can access the API.
  • JWT (JSON Web Tokens) — Self-contained token with a payload (user info, expiry). Passed as Authorization: Bearer <token>. Stateless and widely used for web APIs.
  • OAuth 2.0 — Delegated authorization framework. Users authorize third-party apps without sharing passwords. Used by Google, GitHub, Facebook APIs. Involves access tokens, refresh tokens, and scopes.
  • Basic Auth — Base64-encoded username:password sent in the header. Simple but should only be used over HTTPS.

Testing Tools

ToolTypeKey Features
curlCLIUniversal command-line HTTP client. Installed on all Unix systems. curl -X POST -H "Content-Type: application/json" -d '{"name":"Alice"}' https://api.example.com/users
PostmanGUI (Desktop)Visual API client with collections, environments, test scripts, and code generation. Best for exploration and manual testing.
InsomniaGUI (Desktop)Similar to Postman but more lightweight. Supports GraphQL natively. Open-source.
HTTPieCLIModern curl alternative with nicer output syntax: http POST https://api.example.com/users name=Alice
hurlCLIRun and test API requests from a plain-text file. Great for integration testing.

Best Practices & Versioning

  • Use nouns, not verbs — /api/users, not /api/getUsers. HTTP methods indicate the action.
  • Version your API — Prefix with /v1/, /v2/ in the URL or use the Accept header. Never break existing clients.
  • Consistent naming — Use plural nouns, snake_case or camelCase consistently, and descriptive field names.
  • Pagination — Return large collections in pages: GET /api/users?page=2&limit=50, include total count and next/prev links.
  • Error responses — Always return a consistent error format: {"error": {"code": "VALIDATION_ERROR", "message": "Email is required", "details": [...]}}
  • Use HTTPS always — Never expose an API over plain HTTP. Encrypt everything with TLS.
  • Add rate limiting — Protect your API with rate limits and return 429 Too Many Requests when exceeded.
Pro tip: Use OpenAPI/Swagger to document your API. A well-written OpenAPI spec serves as both documentation and a contract that can generate client SDKs and server stubs.
