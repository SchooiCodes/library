API Design (REST & GraphQL)

API design — RESTful principles, GraphQL, versioning, documentation, and rate limiting.

Table of Contents (2 sections)

  • RESTful Design
  • GraphQL & Best Practices

RESTful Design

REST: Representational State Transfer. Resources: nouns (/users, /articles/{id}), actions via HTTP methods. CRUD: POST (create), GET (read), PUT (replace), PATCH (partial update), DELETE. Status codes: 200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 500 Internal Server Error. Query params: filtering (?status=active), sorting (?sort=-created_at), pagination (?page=1&per_page=20 — cursor-based preferred for large sets). Fields: sparse fieldsets (?fields=id,name,email). HATEOAS: links in responses for discoverability. OpenAPI/Swagger: specification format — OpenAPI Generator, Swagger UI, ReDoc. API versioning: URL path (/v1/users), header (Accept: application/vnd.api+json;version=1), query param.

GraphQL & Best Practices

GraphQL: query language by Meta — single endpoint (/graphql). Schema: type Query { users: [User!]! } type User { id: ID! name: String posts: [Post!]! }. Queries: { users { id name posts { title } } }. Mutations: mutation { createUser(input: {name: "Alice"}) { id } }. Subscriptions: WebSocket-based real-time. Resolvers: functions per field — Dataloader (batching + caching). N+1 problem: solved by DataLoader, JOINs. Fragments: reusable field selections. Apollo: client + server — cache normalization. Rate limiting: token bucket (Leaky Bucket), sliding window — headers X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset. Backend best practices: idempotency keys, proper error responses (RFC 7807 Problem Details), request validation (JSON Schema).
