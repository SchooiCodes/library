🔐 SSH Guide

Master the Secure Shell — keys, config, tunneling, and remote administration.

What is SSH?

SSH (Secure Shell) is a cryptographic network protocol for operating network services securely over an unsecured network. It is most commonly used for remote login to servers, but also supports tunneling, port forwarding, file transfers, and command execution.

SSH uses a client-server architecture. The sshd daemon runs on the server (port 22 by default), and clients connect using the ssh command. All traffic is encrypted end-to-end using public-key cryptography, protecting against eavesdropping, connection hijacking, and other network-level attacks.

Key Generation (ssh-keygen)

SSH key pairs provide password-less authentication that is both more convenient and more secure than passwords:

# Generate an Ed25519 key pair (recommended)
ssh-keygen -t ed25519 -C "your@email.com"

# Generate an RSA key pair (legacy compat)
ssh-keygen -t rsa -b 4096 -C "your@email.com"

# Specify a custom file location
ssh-keygen -t ed25519 -f ~/.ssh/my_server_key -C "my server"

# View the public key (safe to share)
cat ~/.ssh/id_ed25519.pub

# Copy public key to a remote server (one-liner)
ssh-copy-id user@server.com

This generates two files: ~/.ssh/id_ed25519 (private key — never share this!) and ~/.ssh/id_ed25519.pub (public key — safe to share).

Never share your private key. If someone obtains your private key, they can authenticate as you on any server where your public key is installed. Use a strong passphrase and consider using ssh-agent to cache it.

SSH Config (~/.ssh/config)

The SSH config file lets you define shortcuts and per-host settings so you don't have to type long commands each time:

# ~/.ssh/config

# Defaults for all hosts
Host *
  ServerAliveInterval 60
  ServerAliveCountMax 3

# Shortcut for your personal server
Host myserver
  HostName 192.168.1.100
  User alice
  Port 2222
  IdentityFile ~/.ssh/myserver_key

# GitHub (use a specific key for git operations)
Host github.com
  HostName github.com
  User git
  IdentityFile ~/.ssh/github_key

With this config, ssh myserver expands to ssh -p 2222 alice@192.168.1.100 -i ~/.ssh/myserver_key.

authorized_keys — On the server, ~/.ssh/authorized_keys contains one public key per line. Only clients whose private key matches one of these entries can authenticate:

# Manually append a public key to the server
echo "ssh-ed25519 AAAAC3..." >> ~/.ssh/authorized_keys

# Set correct permissions
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys

SSH Tunneling & Port Forwarding

SSH can forward network ports, creating secure tunnels through untrusted networks:

# Local port forwarding: local:8080 -> server:80
ssh -L 8080:localhost:80 user@server

# Remote port forwarding: server:8080 -> local:3000
ssh -R 8080:localhost:3000 user@server

# Dynamic port forwarding (SOCKS proxy on localhost:1080)
ssh -D 1080 user@server

# Jump host: connect to internal server through a bastion
ssh -J jumpuser@bastion.example.com targetuser@internal-server
FlagTypeUse Case
-LLocal forwardAccess a remote service as if it were running locally (e.g., a database behind a firewall)
-RRemote forwardExpose a local service to the remote server (e.g., sharing a dev server)
-DDynamic forwardCreate a SOCKS5 proxy for secure browsing
-JJump hostConnect through an intermediate server (bastion host)

scp & rsync over SSH

Both scp and rsync operate over SSH, providing secure file transfers:

# scp: simple but no resume support
scp file.txt user@server:/home/user/
scp -r myfolder/ user@server:/home/user/
scp user@server:/remote/file.txt ./local/

# rsync: smarter, faster, supports resume
rsync -avz myfolder/ user@server:/home/user/myfolder/
rsync -avz --delete user@server:/source/ ./local/backup/

rsync is preferred for large transfers or backups because it only transfers differences (deltas) and can resume interrupted transfers. Use -z for compression and -P for progress display.

Common Options & Troubleshooting

Flag / OptionDescription
-v, -vv, -vvvVerbose output — each v adds more detail for debugging

Common issues and fixes:

  • "Permission denied (publickey)" — Your public key is not in the server's authorized_keys or the key permissions are wrong. Check chmod 600 ~/.ssh/authorized_keys on the server.
  • "Connection refused" — The SSH daemon is not running on the server, or a firewall is blocking port 22. Check systemctl status sshd and firewall rules.
  • "WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED" — The server's host key changed (reinstall or MITM attack). Clear the old key with ssh-keygen -R hostname.
  • Slow connection — The server may be trying DNS lookups. Set UseDNS no in /etc/ssh/sshd_config on the server, or use -o "GSSAPIAuthentication=no".
Pro tip: Always keep your SSH software updated. Use ssh -V to check your client version and ssh -Q cipher to list supported encryption ciphers.

SSH Agent & Key Management

The SSH agent caches your decrypted private key so you don't have to type your passphrase repeatedly:

# Start the agent in the background
eval "$(ssh-agent -s)"

# Add your key to the agent
ssh-add ~/.ssh/id_ed25519

# List loaded keys
ssh-add -l

# Remove all keys
ssh-add -D

# Forward agent to a remote host (use with caution!)
ssh -A user@server

# Timeout for key caching (macOS)
ssh-add -t 3600 ~/.ssh/id_ed25519  # Cache for 1 hour
Security warning: Agent forwarding (-A) lets the remote server use your local SSH keys. Only use it with servers you trust completely, and consider using -J jump hosts instead where possible.

Practical Examples

Here are real-world SSH workflows you'll use daily:

# Copy your public key to a new server
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@192.168.1.100

# Mount a remote filesystem over SSH (SSHFS)
sshfs user@server:/remote/path /local/mountpoint

# Run a remote command and capture output locally
ssh user@server "uptime && df -h" > server-status.txt

# Create a tunnel to access a remote database locally
ssh -L 5432:localhost:5432 user@db-server

# Use a jump host to reach an internal server
ssh -J bastion.example.com internal-server.local

# Execute a script on multiple servers
for host in server1 server2 server3; do
  ssh user@$host "sudo systemctl restart nginx"
done

SSHFS is particularly useful — it lets you mount a remote directory over SSH so it appears as a local folder. Install with sudo apt install sshfs on Linux or brew install macfuse sshfs on macOS.
