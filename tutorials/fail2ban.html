🛡️ Fail2ban

Protect your Linux server from brute-force attacks with Fail2ban intrusion prevention.

What is Fail2ban?

Fail2ban is an intrusion prevention framework for Linux that protects servers from brute-force attacks. It scans log files for repeated failed authentication attempts and temporarily bans offending IP addresses using firewall rules.

Written in Python, Fail2ban works by monitoring log files (like /var/log/auth.log or /var/log/nginx/access.log) for patterns indicative of attacks. When it detects too many failures from an IP within a configured time window, it updates firewall rules (iptables, nftables, or ufw) to block that IP for a set duration.

Fail2ban ships with filters and jails for over 100 services including SSH, Apache, Nginx, Postfix, Dovecot, vsftpd, and many more.

Installation

Install Fail2ban using your package manager:

# Debian / Ubuntu
sudo apt update
sudo apt install fail2ban

# RHEL / Fedora / CentOS
sudo dnf install fail2ban

# Arch Linux
sudo pacman -S fail2ban

# Enable and start
sudo systemctl enable fail2ban --now

# Check status
sudo systemctl status fail2ban
sudo fail2ban-client status

After installation, Fail2ban runs as a systemd service. Configuration files are in /etc/fail2ban/. Key files: fail2ban.conf (global settings), jail.conf (default jails — do not edit), and jail.local (your custom overrides).

Tip: Never edit jail.conf directly. Create /etc/fail2ban/jail.local with your overrides. Updates will overwrite jail.conf but leave your jail.local unchanged.

How Jails Work

A jail in Fail2ban defines a protective rule. It combines a filter (log pattern), an action (what to do when triggered), and parameters like maxretry, findtime, and bantime:

[sshd]
enabled  = true
port   = ssh
filter  = sshd
logpath  = /var/log/auth.log
maxretry = 5
findtime = 600  # 10 minutes
bantime  = 3600  # 1 hour

This jail monitors auth.log for SSH authentication failures. If an IP fails 5 times within 10 minutes (findtime), it gets banned for 1 hour (bantime).

Configuring jails.local

Create your custom configuration in /etc/fail2ban/jail.local:

[DEFAULT]
# Global settings
bantime = 3600
findtime = 600
maxretry = 5
banaction = iptables-multiport
ignoreip = 127.0.0.1/8 ::1 # Never ban localhost
     192.168.1.0/24   # Your internal network

destemail = admin@example.com
sender = fail2ban@example.com
action = %(action_mwl)s   # Ban and send email with whois

# SSH jail
[sshd]
enabled = true
port = ssh
filter = sshd
logpath = %(sshd_log)s

# SSH with custom port
[sshd-custom]
enabled = true
port = 2222
filter = sshd
logpath = /var/log/auth.log
maxretry = 3
bantime = 86400 # 24 hours for repeat offenders

SSH Protection

SSH is the most common target for brute-force attacks. The default SSH jail works out of the box, but you can tune it:

[sshd]
enabled = true
port = ssh
filter = sshd
logpath = /var/log/auth.log
maxretry = 3
findtime = 300  # 5 minutes
bantime = 86400  # 24 hours

# For high-traffic servers, consider:
# - Longer bantime for repeat offenders
# - Use recidive jail for persistent attackers
# - Fail2ban + SSH key-based auth (disable passwords)

# Check SSH ban status
sudo fail2ban-client status sshd

# Example output:
# Status for the jail: sshd
# |- Filter
# | |- Currently failed: 3
# | |- Total failed: 127
# | `- Currently banned: 5
# `- Actions
#  |- Currently banned: 5
#  |- Total banned: 42
#  `- Banned IP list: 10.0.0.1 10.0.0.2 10.0.0.3

Web Server Protection

Protect Nginx or Apache from web-based attacks with built-in filters:

[nginx-http-auth]
enabled = true
filter = nginx-http-auth
port = http,https
logpath = /var/log/nginx/error.log
maxretry = 5
findtime = 600
bantime = 3600

# Block probes for common paths
[nginx-botsearch]
enabled = true
filter = nginx-botsearch
port = http,https
logpath = /var/log/nginx/access.log
maxretry = 10
findtime = 600
bantime = 86400

# Apache equivalents
[apache-auth]
enabled = true
filter = apache-auth
logpath = /var/log/apache2/*error.log
maxretry = 5

# Prevent WordPress brute force
[wordpress]
enabled = true
filter = wordpress
logpath = /var/log/nginx/access.log
maxretry = 10
findtime = 600
bantime = 3600

Custom Filters & Actions

You can create custom filters for non-standard services. Filters use Python regex to match log patterns:

# /etc/fail2ban/filter.d/myapp.conf
[Definition]
failregex = ^.*Authentication failed for user.*from <HOST>$
ignoreregex =

# Custom action: block with nftables
# /etc/fail2ban/action.d/nftables-custom.conf
[Definition]
actionstart = nft add set inet filter fail2ban { type ipv4_addr\; }
actionban = nft add element inet filter fail2ban { <ip> }
actionunban = nft delete element inet filter fail2ban { <ip> }

# Use it in jail.local:
[myapp]
enabled = true
filter = myapp
logpath = /var/log/myapp/access.log
maxretry = 3
findtime = 300
bantime = 3600
action = nftables-custom

The <HOST> placeholder in regex captures the IP address. Test your filter with fail2ban-regex /var/log/myapp/log "^(.*)<HOST>(.*)$".

Ban Actions & Unbanning

Fail2ban uses actions to enforce bans. Common actions include iptables rules, nftables sets, or cloud API calls:

# View currently banned IPs
sudo fail2ban-client banned
sudo fail2ban-client status sshd

# Unban a specific IP
sudo fail2ban-client set sshd unbanip 192.168.1.100

# Unban all IPs from a jail
sudo fail2ban-client set sshd unbanip --all

# Manually ban an IP
sudo fail2ban-client set sshd banip 10.0.0.99

# Unban across all jails
sudo fail2ban-client unban --all

# Check if an IP is banned
sudo fail2ban-client status sshd | grep "Banned IP list"

In addition to iptables/nftables, Fail2ban can execute custom scripts (e.g., block IPs in Cloudflare firewall), send emails, or log to syslog.

Monitoring with fail2ban-client

The fail2ban-client tool provides real-time monitoring and administration:

# List all active jails
sudo fail2ban-client status

# Detailed status for a specific jail
sudo fail2ban-client status sshd

# Get banned IPs as JSON
sudo fail2ban-client get sshd banned

# Change bantime for a running jail (no restart needed)
sudo fail2ban-client set sshd bantime 7200

# Reload all configuration
sudo fail2ban-client reload

# Reload a single jail
sudo fail2ban-client reload sshd

# Restart Fail2ban
sudo fail2ban-client restart

# View fail2ban log
sudo tail -f /var/log/fail2ban.log

# Test log parsing
sudo fail2ban-regex /var/log/auth.log /etc/fail2ban/filter.d/sshd.conf

Best Practices

  • Create jail.local instead of editing jail.conf — preserves configs across updates.
  • Tune findtime and maxretry based on your risk profile. Lower values are more restrictive but may ban legitimate users.
  • Use the recidive jail — it watches other jails and applies longer bans to persistent offenders (e.g., 1 week).
  • Whitelist trusted IPs with ignoreip in [DEFAULT] to avoid locking out admins.
  • Monitor logs regularly with sudo tail -f /var/log/fail2ban.log to see what's being blocked.
  • Combine with key-based SSH auth — if you disable password auth entirely, SSH brute-force becomes impossible.
  • Set up email alerts with action = %(action_mwl)s to get notified when bans happen.
  • Use fail2ban-client set JAIL bantime SECONDS to adjust penalties on the fly without restarting.
