What is Nginx?

Nginx (pronounced "engine-x") is a high-performance web server, reverse proxy, load balancer, and HTTP cache. It was created by Igor Sysoev in 2004 to solve the C10K problem — handling 10,000+ concurrent connections with limited hardware.

Unlike Apache, Nginx uses an event-driven, asynchronous architecture rather than spawning a thread or process per connection. This makes it extremely efficient for serving static files, proxying to application servers, and handling high-traffic websites.

As of 2025, Nginx powers over 30% of all websites globally, including Netflix, Airbnb, Dropbox, and WordPress.com.

Installation

Nginx is available in all major package repositories:

# Debian / Ubuntu
sudo apt update
sudo apt install nginx

# RHEL / Fedora
sudo dnf install nginx

# Arch Linux
sudo pacman -S nginx

# macOS (Homebrew)
brew install nginx

# Enable and start
sudo systemctl enable nginx --now

After installation, open http://localhost in your browser. You should see the default Nginx welcome page.

Check status: Run sudo systemctl status nginx to verify the service is running. Default document root is /var/www/html/ on Debian-based systems.

Basic Configuration

The main configuration file is /etc/nginx/nginx.conf. Key directives:

user www-data;
worker_processes auto;
pid /run/nginx.pid;

events {
  worker_connections 768;
  multi_accept on;
}

http {
  sendfile on;
  tcp_nopush on;
  tcp_nodelay on;
  keepalive_timeout 65;
  types_hash_max_size 2048;

  include /etc/nginx/mime.types;
  default_type application/octet-stream;

  access_log /var/log/nginx/access.log;
  error_log /var/log/nginx/error.log;

  gzip on;

  include /etc/nginx/conf.d/*.conf;
  include /etc/nginx/sites-enabled/*;
}

Best practice is to keep nginx.conf minimal and put site-specific configs in /etc/nginx/sites-available/ with symlinks to sites-enabled/.

Server Blocks (Virtual Hosts)

Server blocks are Nginx's equivalent of Apache's virtual hosts. Here is a basic site configuration:

server {
  listen 80;
  server_name example.com www.example.com;
  root /var/www/example.com;
  index index.html index.htm;

  location / {
    try_files $uri $uri/ =404;
  }
}

Enable it by creating a symlink:

sudo ln -s /etc/nginx/sites-available/example.com /etc/nginx/sites-enabled/
sudo nginx -t  # Test configuration
sudo systemctl reload nginx

Reverse Proxy

Nginx excels as a reverse proxy, forwarding requests to backend application servers. This is how you'd run a Node.js, Python, or Java app behind Nginx:

server {
  listen 80;
  server_name app.example.com;

  location / {
    proxy_pass http://127.0.0.1:3000;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection 'upgrade';
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_cache_bypass $http_upgrade;
  }
}

This forwards all requests to a backend running on port 3000 (e.g., a Node.js app). WebSocket support is included via the Upgrade header.

Load Balancing

Distribute traffic across multiple backend servers for high availability:

upstream backend {
  least_conn; # Send to server with fewest connections
  server 10.0.0.1:3000 weight=3;
  server 10.0.0.2:3000;
  server 10.0.0.3:3000 backup;
}

server {
  listen 80;
  server_name loadbalanced.app;

  location / {
    proxy_pass http://backend;
  }
}

Load balancing methods include round-robin (default), least_conn, ip_hash (session persistence), and random.

SSL / HTTPS

Use Let's Encrypt with Certbot for free SSL certificates:

sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d example.com -d www.example.com

A typical HTTPS server block:

server {
  listen 443 ssl http2;
  server_name example.com;

  ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
  ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;
  ssl_protocols TLSv1.2 TLSv1.3;
  ssl_ciphers HIGH:!aNULL:!MD5;

  root /var/www/example.com;
  index index.html;

  location / {
    try_files $uri $uri/ =404;
  }
}

# Redirect HTTP to HTTPS
server {
  listen 80;
  server_name example.com www.example.com;
  return 301 https://$server_name$request_uri;
}

Static File Server

Nginx can serve static files with exceptional performance:

server {
  listen 80;
  server_name files.example.com;
  root /var/www/files;

  # Cache static assets
  location ~* \.(jpg|jpeg|png|gif|ico|css|js)$ {
    expires 30d;
    add_header Cache-Control "public, immutable";
  }

  # Directory listing
  location /downloads {
    autoindex on;
    autoindex_exact_size off;
    autoindex_localtime on;
  }

  # Prevent access to hidden files
  location ~ /\. {
    deny all;
  }
}

Security Hardening

Essential security configurations for production Nginx:

# Hide Nginx version
server_tokens off;

# Limit request size (prevents DOS attacks)
client_max_body_size 10M;

# Clickjacking protection
add_header X-Frame-Options "SAMEORIGIN" always;

# XSS protection
add_header X-XSS-Protection "1; mode=block" always;

# MIME-type sniffing protection
add_header X-Content-Type-Options "nosniff" always;

# HSTS (if using HTTPS)
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

# Rate limiting
limit_req_zone $binary_remote_addr zone=mylimit:10m rate=10r/s;
location /login {
  limit_req zone=mylimit burst=20;
  proxy_pass http://backend;
}

# Block common exploits
location ~* (eval|base64_encode|base64_decode) {
  deny all;
}

Troubleshooting

Common Nginx issues and how to fix them:

  • Test config before reload: Always run sudo nginx -t before systemctl reload nginx to avoid locking yourself out.
  • Check logs: /var/log/nginx/access.log and /var/log/nginx/error.log are your first debugging tools.
  • Permission denied: Ensure www-data (or your nginx user) has read access to the web root directory.
  • Port already in use: Another web server (Apache?) may be running. Check with sudo ss -tlnp | grep 80.
  • 502 Bad Gateway: Your backend server is not running or unreachable. Check the application and the proxy_pass target.
  • Reload vs restart: Use reload for config changes (zero-downtime), restart only when the process itself needs restarting.
