What is GitHub Actions?

GitHub Actions is a CI/CD (Continuous Integration and Continuous Deployment) platform that automates your software workflows directly from your GitHub repositories. You can build, test, lint, package, and deploy your code right where it lives.

Actions uses YAML-based workflow files stored in .github/workflows/. Each workflow is triggered by events like pushes, pull requests, or scheduled times. GitHub provides free minutes for public repositories and a generous allotment for private ones.

With over 15,000 community actions in the Marketplace, you can integrate virtually any tool or service without writing everything from scratch.

Workflows & YAML

A workflow is a configurable automated process defined in a YAML file. Here is the simplest possible workflow:

name: CI

on: [push]

jobs:
 build:
  runs-on: ubuntu-latest
  steps:
   - uses: actions/checkout@v4
   - run: echo "Hello, world!"

Workflow files live in .github/workflows/ and can have any name (e.g., ci.yml, deploy.yml). You can have multiple workflows in the same repository for different purposes.

Key YAML keys: name (display name), on (trigger event), jobs (work units), runs-on (runner type), and steps (individual tasks).

Tip: Use workflow_dispatch: under on to allow manual triggering of workflows from the GitHub UI.

Jobs & Steps

Jobs are units of work that run on the same runner. Steps are individual tasks within a job. Jobs can run in parallel or sequentially with needs dependencies:

jobs:
 lint:
  runs-on: ubuntu-latest
  steps:
   - uses: actions/checkout@v4
   - uses: actions/setup-node@v4
    with:
     node-version: 20
   - run: npm ci
   - run: npm run lint

 test:
  needs: lint
  runs-on: ubuntu-latest
  steps:
   - uses: actions/checkout@v4
   - uses: actions/setup-node@v4
    with:
     node-version: 20
   - run: npm ci
   - run: npm test

 deploy:
  needs: [lint, test]
  runs-on: ubuntu-latest
  steps:
   - uses: actions/checkout@v4
   - run: npm run build
   - run: ./deploy.sh

Use needs to create dependency chains. Jobs without needs run in parallel by default.

Runners & Environments

Runners are the servers that execute your workflows. GitHub provides hosted runners (Ubuntu, Windows, macOS) and you can host your own self-hosted runners:

jobs:
 test:
  runs-on: ubuntu-latest # GitHub-hosted

 deploy:
  runs-on: self-hosted  # Your own machine
  environment: production
  steps:
   - run: ./deploy.sh

Environments provide deployment targets with protection rules. You can require approvals before deploying to production, limit which branches can deploy, and track deployments with timestamps.

Self-hosted runners are useful when you need specific hardware, on-premises access, or want to avoid per-minute costs.

Triggers & Events

Workflows are triggered by events. The on key supports many event types:

on:
 push:
  branches: [main, develop]
  paths-ignore:
   - '**.md'
 pull_request:
  branches: [main]
  types: [opened, synchronize, reopened]
 schedule:
  - cron: '0 2 * * *' # Daily at 2 AM
 workflow_dispatch:
  inputs:
   environment:
    description: 'Deploy target'
    required: true
    default: 'staging'
 release:
  types: [published]

Use branches, tags, and paths filters to control exactly when workflows run. The schedule event uses POSIX cron syntax.

Actions Marketplace

The GitHub Marketplace hosts thousands of community and official actions. Instead of writing bash scripts, you compose actions like building blocks:

steps:
 # Official actions
 - uses: actions/checkout@v4
 - uses: actions/setup-python@v5
  with:
   python-version: '3.12'

 # Community actions
 - uses: docker/login-action@v3
  with:
   username: ${{ secrets.DOCKER_USER }}
   password: ${{ secrets.DOCKER_TOKEN }}

 - uses: docker/build-push-action@v5
  with:
   push: true
   tags: myapp:latest

Popular actions include actions/checkout, actions/setup-node, actions/cache, docker/build-push-action, aws-actions/configure-aws-credentials, and rtCamp/action-slack-notify.

Always pin actions to a Git SHA (e.g., @v4 is a major version tag, but @abc123def is immutable) for security and reproducibility.

Matrix Builds

Matrix builds let you test across multiple OS versions, language versions, and configurations in parallel:

jobs:
 test:
  runs-on: ${{ matrix.os }}
  strategy:
   matrix:
    os: [ubuntu-latest, windows-latest, macos-latest]
    node: [18, 20, 22]
    include:
     - os: ubuntu-latest
      node: 22
      coverage: true
   fail-fast: false

  steps:
   - uses: actions/checkout@v4
   - uses: actions/setup-node@v4
    with:
     node-version: ${{ matrix.node }}
   - run: npm ci
   - run: npm test
   - if: matrix.coverage
    run: npm run coverage

Matrix builds generate every combination of the variables. Use include to add extra variables for specific combinations and exclude to skip certain combos.

Secrets & Variables

Secrets are encrypted environment variables for sensitive data. Variables are plaintext values for non-sensitive configuration:

jobs:
 deploy:
  runs-on: ubuntu-latest
  environment: production
  steps:
   - run: ./deploy.sh
    env:
     API_KEY: ${{ secrets.API_KEY }}
     DB_PASSWORD: ${{ secrets.DB_PASSWORD }}
     DEPLOY_ENV: ${{ vars.DEPLOY_ENVIRONMENT }}

Set secrets in Settings > Secrets and variables > Actions in your repository or organization. Secrets are masked in logs and never exposed to forks. Use vars context for non-sensitive config like deployment URLs or feature flags.

Caching Dependencies

Cache dependencies to speed up workflows by avoiding repeated downloads:

steps:
 - uses: actions/checkout@v4

 - name: Cache Node modules
  id: cache-npm
  uses: actions/cache@v4
  with:
   path: ~/.npm
   key: ${{ runner.os }}-node-${{ hashFiles('**/package-lock.json') }}
   restore-keys: |
    ${{ runner.os }}-node-

 - if: steps.cache-npm.outputs.cache-hit != 'true'
  run: npm ci

The cache action uses a key based on a hash of lockfiles to invalidate when dependencies change. Use restore-keys as a fallback to partially restore caches. Common cache targets: ~/.npm, ~/.cache/pip, ~/.gradle/caches, ~/.m2/repository.

Cache size is limited to 10 GB per repository. Use the gh actions-cache CLI to manage and prune old entries.

Best Practices

  • Pin action versions to commit SHAs for security. Major version tags (@v4) can be overwritten.
  • Use minimal permissions with permissions: block. Don't use the default GITHUB_TOKEN with write access unless needed.
  • Keep workflows fast by caching dependencies, using matrix builds wisely, and parallelizing independent jobs.
  • Use if conditions to skip steps conditionally: if: github.ref == 'refs/heads/main' for deploy-only steps.
  • Test workflow changes on a branch before merging to main. Use workflow_dispatch to test manually.
  • Use actions/upload-artifact to persist build outputs between jobs and actions/download-artifact to retrieve them.
  • Set a timeout for every job with timeout-minutes: 30 to prevent runaway workflows from consuming all minutes.
