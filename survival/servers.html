Emergency Local Servers & Mesh Networking

Keep your local network running when the internet goes dark. Local DNS, file sharing, mesh messaging, portable web servers, and off-grid communication for disaster scenarios.

Why Local Servers Matter

When the internet goes down — from ISP failure, DNS attacks, natural disaster, or government-imposed blackout — a local network with the right servers keeps your devices useful:

  • DNS filtering — Block ads and malware even when upstream DNS fails. Pi-hole and AdGuard Home cache DNS responses for faster lookup.
  • File sharing — Access files across all your local devices without cloud storage. Samba and NFS let Windows, macOS, Linux, Android, and smart TVs share files.
  • Mesh networking — Communicate with others nearby without any infrastructure. Briar (Bluetooth/Wi-Fi direct) and Meshtastic (LoRa radio) keep messages flowing.
  • Local web servers — Host documentation, wikis, ZIM archives, or development environments on a portable machine. Accessible from any device on the LAN.
  • Low-power operation — A Raspberry Pi 4/5 running all these services draws ~5-15W. A portable battery pack can run it for hours.
Prep now: Set up these services while you have internet. Configure Pi-hole, create Samba shares, install Briar on your phone. When the network goes down, you'll be glad everything is already configured.

Pi-hole — DNS Ad Blocking

Pi-hole is a network-wide ad blocker that acts as your local DNS server. It blocks ads, trackers, and malware domains at the DNS level — before they even load. Works across all devices on your network.

  • How it works: Pi-hole intercepts DNS queries from every device on your network. If a domain is on its blocklist, it returns a null response (the ad never loads).
  • Benefits during outage: Even without internet, Pi-hole caches local DNS names. It also provides a DHCP server so devices get IP addresses automatically.
  • Hardware: Runs on Raspberry Pi, any Linux machine, Docker, or even in a VM. Minimal requirements: 512MB RAM, 2GB storage.
# Install Pi-hole (one-line script)
curl -sSL https://install.pi-hole.net | bash

# During setup:
# - Choose a static IP address (e.g., 192.168.1.100)
# - Select DNS upstream provider (Cloudflare, Google, Quad9, etc.)
# - Enable DHCP server (optional but recommended)
# - Choose blocklists (default is good for most users)

# After installation:
# Web admin: http://pi.hole/admin/ (password set during install)
# pihole -g         # Update gravity (blocklists)
# pihole -a -p        # Change web password
# pihole -b example.com   # Whitelist a domain
# pihole -d example.com   # Blacklist a domain

# Set Pi-hole as DNS on your router:
# Router settings → DHCP → DNS Server → 192.168.1.100
# Or set manually on each device

With Pi-hole running, your network gets ad-free browsing, DNS caching (faster lookups), and protection from known malware domains. During an internet outage, local DNS still works for local resources.

Pro tip: When internet is out, Pi-hole still caches previously resolved domains. This means frequently visited sites that are cached may still load partial content from your browser cache. Combined with Kiwix Serve, your LAN becomes a self-contained information system.

AdGuard Home — Full DNS Suite

AdGuard Home is a modern alternative to Pi-hole with a cleaner interface and more built-in features. It provides DNS filtering, DHCP, parental controls, and detailed statistics in a single package.

  • Differences from Pi-hole: Built-in DNS-over-HTTPS/DNS-over-TLS, encrypted DNS upstream, query log with filters, client-specific settings, more polished web UI.
  • Installation: Binary, Docker, or direct package install. Supports Linux, macOS, Windows, and BSD.
  • Performance: Go-based — very fast, low memory usage. Handles high query volumes easily.
# Install AdGuard Home via Docker
docker run --name adguardhome \
 --restart unless-stopped \
 -v /opt/adguard/work:/opt/adguardhome/work \
 -v /opt/adguard/conf:/opt/adguardhome/conf \
 -p 53:53/tcp -p 53:53/udp \
 -p 80:80/tcp -p 443:443/tcp \
 -d adguard/adguardhome:latest

# Access web UI: http://localhost:80
# Setup wizard: configure upstream DNS, enable DHCP, set filters

# Direct install (Linux):
curl -sSL https://raw.githubusercontent.com/AdguardTeam/AdGuardHome/master/scripts/install.sh | sh

# Configure as system DNS:
# Same as Pi-hole — point your router's DNS to this machine's IP

Both Pi-hole and AdGuard Home serve the same primary purpose. Choose Pi-hole for maturity and massive community, AdGuard Home for a more polished modern interface and encrypted DNS support.

Samba — Cross-Platform File Sharing

Samba implements the SMB/CIFS protocol, allowing Windows, macOS, Linux, Android, and even smart TVs to share files over the local network. No cloud services needed.

  • Key use: Central file server for your emergency documents, ZIM archives, software installers, and backups. Accessible from any device on the LAN.
  • Performance: Gigabit Ethernet = ~110 MB/s. Wi-Fi 5/6 = ~40-80 MB/s. Good enough for media streaming and file access.
# Install Samba
sudo apt install samba      # Debian/Ubuntu
sudo pacman -S samba       # Arch

# Configure /etc/samba/smb.conf
[shared]
 path = /srv/samba/shared
 browseable = yes
 read only = no
 guest ok = yes
 create mask = 0755

[backups]
 path = /srv/samba/backups
 browseable = yes
 read only = yes
 guest ok = no
 valid users = @backup-users

# Create directories and set permissions
sudo mkdir -p /srv/samba/{shared,backups}
sudo chmod 777 /srv/samba/shared
sudo chown root:backup-users /srv/samba/backups

# Start Samba
sudo systemctl enable smbd
sudo systemctl start smbd

# Access from Windows: \\server-ip\shared
# Access from macOS: Finder → Go → Connect to Server → smb://server-ip/shared
# Access from Linux: nautilus smb://server-ip/shared
# Access from Android: CX File Explorer → Network → SMB

For emergency scenarios, set up a Samba share with guest access (no password) on the "shared" folder. This lets anyone on the local network access critical documents and tools.

NFS — Linux Network File System

NFS is the native file sharing protocol for Unix/Linux systems. It's simpler than Samba and offers better performance for Linux-to-Linux file sharing.

  • When to use: Linux-only networks, high-performance requirements, or when you need POSIX permissions preserved.
  • Versions: NFSv3 (older, works everywhere), NFSv4 (newer, better security, single port). Use NFSv4 if possible.
# Install NFS server
sudo apt install nfs-kernel-server  # Debian/Ubuntu
sudo pacman -S nfs-utils       # Arch

# Configure /etc/exports
/srv/nfs/shared *(rw,sync,no_subtree_check,no_root_squash)
/srv/nfs/backups 192.168.1.0/24(ro,sync,no_subtree_check)

# Create export directory and apply config
sudo mkdir -p /srv/nfs/{shared,backups}
sudo exportfs -a
sudo systemctl enable nfs-server
sudo systemctl start nfs-server

# Mount from client:
sudo mount -t nfs4 server-ip:/srv/nfs/shared /mnt/shared
# Or add to /etc/fstab:
server-ip:/srv/nfs/shared /mnt/shared nfs4 defaults 0 0

For a mixed-OS network, run both Samba and NFS. Samba for Windows/macOS clients, NFS for Linux clients. Both can share the same directories.

Briar — Off-Grid Messaging

Briar is a messaging app designed for activists, journalists, and anyone who needs to communicate when internet is unavailable or unsafe. It works over Bluetooth, Wi-Fi Direct, or Tor.

  • No internet required: Briar creates direct connections between nearby devices via Bluetooth or Wi-Fi Direct. Messages sync when devices are in range.
  • End-to-end encrypted: All messages are encrypted. Briar uses the Bramble handshake protocol and has forward secrecy.
  • Tor support: When internet is available, Briar routes messages through Tor for anonymity.
# Install Briar (Android only currently)
# F-Droid: https://f-droid.org/packages/org.briarproject.briar.android/
# Google Play: available but use F-Droid for full privacy

# Getting started:
# 1. Create account (no phone number or email required)
# 2. Add contacts via:
#  - Bluetooth discovery (nearby)
#  - QR code scanning (in person)
#  - Invite link (over any available channel)
# 3. Start messaging — messages sync when devices are near

# Off-grid use case:
# 1. Two people in same building install Briar
# 2. Connect via Bluetooth (range ~10m) or Wi-Fi Direct (range ~50m)
# 3. Messages sync automatically when in range
# 4. Forum and blog features for group communication

Limitations: Android-only at the time of writing. iOS version is in development. Messages are not relayed — both devices must be in range simultaneously for delivery.

Use with Meshtastic: Briar for rich messaging (text, images, forums) within ~10-50m range. Meshtastic for long-range text-only communication over kilometers. Together they cover short and long-range off-grid messaging.

Meshtastic — LoRa Mesh Networking

Meshtastic creates a long-range, low-power mesh network using LoRa radio modules. Each device acts as a node, forwarding messages to others. Can cover kilometers with multiple nodes.

  • Hardware: LoRa radios (Heltec Lora 32, TTGO T-Beam, RAK WisBlock, LilyGO). ~$20-50 per device. Use with a phone via Bluetooth.
  • Range: 1-5 km per node in urban areas, 5-15 km line-of-sight. Each node extends the mesh.
  • Channels: Multiple channels for different groups. Primary channel is encrypted. Custom channels for specific purposes.
  • Power: Runs for days on a single battery charge (18650 Li-ion). Solar charging support.
# Flash Meshtastic firmware to LoRa device
# Download flasher from https://flasher.meshtastic.org/
# Connect device via USB, select firmware, flash

# Configure via phone app (iOS/Android) or CLI:
pip install meshtastic       # Python CLI tool

# Set node name and region
meshtastic --set-owner "Emergency-HQ"
meshtastic --set lora.region US   # Set your regulatory region

# Create a primary channel
meshtastic --ch-add primary
# Share the channel QR code with other users

# Check mesh status
meshtastic --info          # Node info and neighbors

# Send message
meshtastic --send "Testing mesh"
meshtastic --send "Help needed" --dest "!c0ffeebabe"

Perfect for: Emergency response teams, camping groups, neighborhood watch, disaster relief, any scenario where cell towers are down and you need text communication over long distances.

Yggdrasil — IPv6 Mesh Networking

Yggdrasil is an overlay network that implements an IPv6 mesh. It's designed to be decentralized, self-organizing, and work over any existing network infrastructure — or none at all.

  • How it works: Every node gets a unique IPv6 address from a cryptographic key pair. Nodes discover each other and form a mesh. Traffic is encrypted end-to-end.
  • No central server: Yggdrasil is fully peer-to-peer. Any node can join and traffic automatically finds the best path.
  • Overlay or direct: Works over the internet (as an overlay) or over direct links (Ethernet, Wi-Fi ad-hoc, WireGuard tunnels).
# Install Yggdrasil
sudo apt install yggdrasil     # Debian/Ubuntu (if in repos)
# Or download from https://github.com/yggdrasil-network/yggdrasil-go

# Generate configuration
yggdrasil -genconf > /etc/yggdrasil/yggdrasil.conf

# Connect to peers (add to config under Peers section)
# Find public peers: https://publicpeers.neilalexander.dev/
# Example:
# Peers:
#  - tcp://1.2.3.4:12345
#  - tls://5.6.7.8:45678

# Start Yggdrasil
sudo systemctl enable yggdrasil
sudo systemctl start yggdrasil

# Your Yggdrasil IP is shown in logs:
# journalctl -u yggdrasil | grep "Your address is"

# Ping another Yggdrasil node:
ping6 201:abcd:ef01:2345::1

# Direct peer over Wi-Fi ad-hoc:
# Set up ad-hoc Wi-Fi network, then add peer:
# Peers:
#  - tcp://192.168.2.2:12345

Use case: If you have a group of devices (laptops, Raspberry Pis) that need a private, encrypted network without any configuration. Yggdrasil auto-configures IP addresses and routing — just install and add peers.

Portable Web Servers

A lightweight web server can serve documentation, ZIM archives via Kiwix, local wikis, or development environments. These options are ideal for emergency local hosting:

  • Kiwix Serve — Serves ZIM archives (already covered in Offline Wikis page). Start with: kiwix-serve --port=8080 *.zim
  • Caddy — Single binary web server with automatic HTTPS. Perfect for serving static files and reverse proxy. Go-based, no dependencies.
  • Python http.server — Zero-setup file server: python3 -m http.server 8000
  • nginx — Full-featured production web server. Configuration-heavy but supports PHP, reverse proxy, load balancing.
  • lighttpd — Lightweight web server optimized for speed. Good for embedded systems (routers, Raspberry Pi).
# Caddy — simplest portable web server
# Download: https://caddyserver.com/download (single binary)

# Serve current directory on port 8080:
caddy file-server --listen :8080 --root /path/to/files

# Caddyfile for reverse proxy with Kiwix:
localhost:8080 {
 reverse_proxy localhost:3000
}

# Python one-liner (no install needed, built-in):
python3 -m http.server 8000
# Serves the current directory at http://0.0.0.0:8000

# nginx static file server (quick config):
# /etc/nginx/sites-enabled/survival
server {
 listen 80;
 server_name survival.local;
 root /var/www/survival;
 index index.html;
 autoindex on;
}

# lighttpd on OpenWrt router:
# opkg install lighttpd
# /etc/lighttpd/lighttpd.conf
server.document-root = "/mnt/usb/www/"
Recommendation: Keep the Caddy binary and a Python interpreter on your emergency USB. Together they cover 99% of local web serving needs — Caddy for production-like setups, Python for quick ad-hoc sharing.

Building Your Emergency Local Network

Putting it all together — here's how to build a complete emergency local network that functions independently of the internet:

Hardware Setup

  • Core server: Raspberry Pi 4/5 (or any small Linux box). 8GB RAM recommended. Boot from SSD, not SD card.
  • Power: 20,000 mAh USB battery pack + UPS for the router. A low-power router (TP-Link WR841, ~5W) can run for hours on a small UPS.
  • Storage: 256GB+ USB SSD or SD card. Store ZIM archives, software repositories, documentation, and backups.
  • Network: Wi-Fi router with LAN ports. Configure with a static LAN IP (192.168.1.1). Set DHCP to hand out addresses in the 192.168.1.100-200 range.
  • Off-grid comms: Meshtastic LoRa devices (2+), Briar on Android phones.

Software Stack

# Emergency server setup script (run while internet works)
#!/bin/bash

# Install base services
apt update
apt install -y samba nfs-kernel-server docker.io

# Install Pi-hole
curl -sSL https://install.pi-hole.net | bash

# Set up Samba share for shared docs
mkdir -p /srv/samba/{shared,public,zim}
chmod 777 /srv/samba/shared
chmod 777 /srv/samba/public
chmod 755 /srv/samba/zim

# Deploy Kiwix with ZIM files
docker run -d --name kiwix \
 -v /srv/samba/zim:/data:ro \
 -p 8080:8080 \
 kiwix/kiwix-serve:latest

# Deploy Wiki.js for local documentation
docker run -d --name wikijs \
 -p 3000:3000 \
 -v /srv/wiki:/data \
 requarks/wiki:latest

echo "Emergency server setup complete!"
echo "Kiwix: http://$(hostname -I | awk '{print $1}'):8080"
echo "Wiki: http://$(hostname -I | awk '{print $1}'):3000"
echo "Pi-hole: http://$(hostname -I | awk '{print $1}')/admin"
