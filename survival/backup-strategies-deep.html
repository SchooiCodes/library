Advanced Backup Strategies

Deep dive into backup strategies: 3-2-1 rule, local and offsite backups, cloud options, rotation schemes, restore testing, and automation.

3-2-1 Rule

3 copies of data, 2 different media types, 1 offsite. Example: laptop (primary) + external HDD (local) + Backblaze (cloud).

Protects against hardware failure, theft, fire, flood, and ransomware.

Local Backups

Fastest for restores.

Options: External USB HDD (4-8 TB, ~$80-120), External USB SSD (1-2 TB, ~$60-120), NAS (~$200-600).

# rsync (Linux/macOS):
rsync -avh --delete /home/user/Documents/ /mnt/backup/
# Robocopy (Windows):
robocopy C:\Users\Name\ D:\Backup\ /MIR

Offsite Backups

Methods: Swapped drives with trusted person (free, monthly swap), safe deposit box, cloud backup, work/office location.

Geographic distance: At least 10 miles away.

Cloud Backups

Providers: Backblaze (~$9/mo unlimited), iDrive (~$80/yr 5 TB), Proton Drive (zero-access E2EE), Rclone + any cloud.

Important: Cloud backup is NOT cloud sync. Backup keeps version history, sync propagates deletions.

Backup Rotation

GFS scheme: 7 daily + 4 weekly + 12 monthly backups. Long retention with deduplication via hardlinks.

Testing Restores

A backup you have not tested is not a backup.

Monthly: Restore 3 random files, verify checksums. Quarterly: Restore entire OS. Yearly: Full disaster recovery drill.

Automation

The best backup is automatic. Use systemd timers (Linux), Task Scheduler (Windows), or cron. Monitor with healthcheck.io.

RTO (Recovery Time Objective) and RPO (Recovery Point Objective)

Two critical metrics define your backup strategy:

  • RPO (Recovery Point Objective) — Maximum acceptable data loss measured in time. "How much data can I afford to lose?" RPO of 1 hour = hourly backups. RPO of 24 hours = daily backups.
  • RTO (Recovery Time Objective) — Maximum acceptable downtime. "How fast do I need to be back online?" RTO of 4 hours = need rapid restore capability. RTO of 48 hours = can use slower methods.
  • Examples: Personal desktop: RPO=24h, RTO=48h. Business server: RPO=1h, RTO=4h. Photo collection: RPO=1 week, RTO=1 month.

Define your RPO and RTO before choosing backup tools. A $10/month cloud backup is fine for 24h RPO. For 1-hour RPO you need local snapshots or continuous backup.

rclone for Cloud Backups

rclone is the Swiss Army knife of cloud storage — it syncs to 40+ providers with encryption:

# rclone — Back up to multiple cloud providers:
# 1. Configure a remote:
rclone config
# → name: mybackblaze
# → type: b2 (Backblaze B2)
# → Follow prompts for key/account

# 2. Back up a directory:
rclone sync /home/user/Documents mybackblaze:backup-bucket/Documents
# --progress shows transfer speed and ETA
# --dry-run shows what would be copied without actually copying

# 3. Encrypt before upload:
rclone config
# → name: cryptremote
# → type: crypt
# → remote: mybackblaze:backup-bucket
# → password: [strong password]
# Files are encrypted client-side before upload

# 4. Verify backups with checksums:
rclone check mybackblaze:backup-bucket/Documents /home/user/Documents

# 5. List files in cloud:
rclone ls mybackblaze:backup-bucket/

# Key flags:
# --backup-dir=/old : move changed files to "old" directory
# --delete-excluded : delete excluded files from destination
# --exclude="*.tmp" : skip temporary files

rclone supports: Backblaze B2, Wasabi, S3, Google Drive, Dropbox, OneDrive, SFTP, WebDAV, and 30+ more.

Restic Configuration Examples

Restic creates encrypted, deduplicated snapshots. Each backup is a point-in-time snapshot stored in a repository:

# Restic — Initialize and back up to multiple backends:

# Backblaze B2:
export B2_ACCOUNT_ID="your_id"
export B2_ACCOUNT_KEY="your_key"
restic -r b2:bucket-name:restic-repo init
restic -r b2:bucket-name:restic-repo backup /home/user/Documents

# Local directory:
restic -r /mnt/backup/restic-repo init
restic -r /mnt/backup/restic-repo backup /home/user --exclude="*.tmp"

# SFTP (NAS or remote server):
restic -r sftp:user@nas:/backup/restic init
restic -r sftp:user@nas:/backup/restic backup /home/user

# AWS S3 (or S3-compatible like Minio):
export AWS_ACCESS_KEY_ID="key"
export AWS_SECRET_ACCESS_KEY="secret"
restic -r s3:s3.amazonaws.com/bucket/restic backup /home/user

# Restic commands:
restic snapshots         # List all snapshots
restic restore latest    # Restore latest snapshot
restic restore 6a1b2c3d  # Restore specific snapshot
restic check             # Verify repository integrity
restic forget --keep-daily 7 --keep-weekly 4
# Clean up old snapshots while keeping 7 daily + 4 weekly

Restic vs Borg: Restic is easier to set up, supports more backends natively, and has better cross-platform support. Borg is more storage-efficient for similar data.

Borg Backup Deduplication

Borg is renowned for its excellent deduplication and compression. Perfect for backing up multiple machines or large datasets with many duplicates:

# Borg — Initialize and back up:
borg init --encryption=repokey-blake2 /mnt/backup/borg-repo
borg create --stats --progress \
    /mnt/backup/borg-repo::backup-{now:%Y-%m-%d} \
    /home/user/Documents \
    --exclude '*.cache' \
    --exclude '*.tmp'

# Borg key features:
borg list /mnt/backup/borg-repo      # List archives
borg extract /mnt/backup/borg-repo::backup-2026-05-01
borg info /mnt/backup/borg-repo::backup-2026-05-01
borg check /mnt/backup/borg-repo     # Verify integrity

# Auto-pruning with borg prune:
borg prune --keep-daily=7 --keep-weekly=4 \
    --keep-monthly=6 /mnt/backup/borg-repo
Deduplication win: Borg splits files into chunks and stores each chunk only once. Backing up 10 VMs that share the same OS base may use only 20% of the space of individual backups.

FreeFileSync GUI Walkthrough

FreeFileSync provides a graphical interface for comparing and synchronizing folders. Ideal for users who prefer a visual workflow:

# FreeFileSync Setup:
# 1. Download FreeFileSync Portable to your USB
# 2. Run FreeFileSync.exe
# 3. Left panel: Browse to source folder (C:\Users\You\Documents)
# 4. Right panel: Browse to destination (D:\Backup\Documents)
# 5. Click "Compare" — shows differences in color:
#    Green = new file, Red = deleted, Yellow = modified
# 6. Choose sync method:
#    - Mirror: Destination matches source exactly
#    - Update: Copy new/modified files (does not delete)
#    - Two-way: Both directions, newest file wins
# 7. Click "Synchronize" — preview changes first, then execute
# 8. Save configuration as .ffs_gui file for repeat use
# 9. Use FreeFileSync Batch (.ffs_batch) for automated runs:
#    FreeFileSync.exe MyBackup.ffs_batch

Windows File History Setup

Windows File History provides continuous versioned backups of user files with minimal configuration:

# Windows File History Setup:
# Settings → Update & Security → Backup → Add a drive
# Select external HDD or network location
#
# Advanced settings:
# - Keep backups: Forever (default) or Until space needed
# - Backup frequency: Every 10 minutes to Daily
# - Cache size: Up to 10% of drive
# - Exclude folders: Add temp, downloads, cache
#
# Restore files:
# Right-click file → Restore previous versions
# Or: Settings → Backup → More options → Restore files
#
# File History stores backups in a FileHistory folder
# on the target drive. Each version is deduplicated
# at the file level.

Veeam Agent for Windows/Linux

Veeam Agent provides full system image backups (bare-metal restore capable):

# Veeam Agent for Windows:
# 1. Download Veeam Agent for Microsoft Windows FREE
# 2. Install → Choose "Free" license
# 3. Backup mode: "Entire computer" for bare-metal restore
#    or "Volume level backup" for specific drives
# 4. Destination: External HDD, NAS, or Veeam Backup Repository
# 5. Schedule: Daily at 2 AM (or custom)
# 6. Retention: 7 restore points by default
# 7. Advanced: Enable encryption (AES-256)
#
# Veeam Agent for Linux:
# 1. Install:
sudo wget -O- https://repository.veeam.com/keys/veeam.asc | apt-key add -
sudo apt install veeam
# 2. Configure backup:
veeamconfig backup create --name "SystemBackup" \
    --include / --destination /mnt/backup
# 3. Schedule:
veeamconfig job create --name "DailyBackup" \
    --backup "SystemBackup" --schedule daily --at 02:00
# 4. Restore entire system:
# Boot from Veeam Recovery Media ISO
# → Select backup file → Restore entire system

Incremental vs Differential vs Full Backup

Type Data Backed Up Restore Speed Storage Used Typical Use
FullAll selected filesFastest (one file)Most (every file)Weekly/monthly baseline
IncrementalChanges since last backupSlowest (must replay all)Least (only changes)Daily/hourly backups
DifferentialChanges since last FULLMedium (full + latest diff)Medium (grows until full)Weekly backups

Recommendation: Full backup weekly + incremental daily. This balances storage, backup speed, and restore speed.

Backup Encryption Standards

Encrypting backups protects your data if the backup media is lost or stolen:

  • AES-256 — Industry standard. Used by VeraCrypt, Restic, Borg, Veeam. Considered unbreakable with current technology.
  • ChaCha20-Poly1305 — Modern stream cipher. Used by Restic (default for new repos). Faster on devices without AES hardware acceleration.
  • GPG encryption — For file-level encryption. Combine with tar: tar czf - /data | gpg -c > backup.tar.gz.gpg
  • End-to-end encryption — Encrypt client-side before uploading to cloud. The provider cannot read your data. Restic, Borg, and Cryptomator all support this.
  • Key management — Store encryption keys separately from backups. If you lose the key, the backup is worthless. Use a password manager for keys.

Backup Testing Procedures

A backup you have not tested is not a backup. Here is a systematic testing approach:

# Monthly Quick Test (30 min):
# 1. Pick 3 random files from backup
# 2. Restore to a test folder
# 3. Verify checksums match originals:
certutil -hashfile restored-file.txt SHA256
certutil -hashfile original-file.txt SHA256
# 4. Open and inspect the files visually

# Quarterly Semi Test (2 hours):
# 1. Restore an entire folder tree
# 2. Verify file count matches:
dir /s /a-d | find /c ":"     # Windows
find . -type f | wc -l         # Linux
# 3. Boot from a recovery ISO
# 4. Mount backup and verify it is readable

# Annual Full Test (1 day):
# 1. Wipe a test machine or VM
# 2. Boot from recovery media
# 3. Perform full bare-metal restore
# 4. Verify all applications work
# 5. Time the restore — does it meet your RTO?

Automated Backup Schedules

Automation is the secret to reliable backups. Here are schedule templates for different tools:

# Linux — systemd timer for Borg backup:
# /etc/systemd/system/borg-backup.service:
[Unit]
Description=Borg backup
[Service]
Type=oneshot
ExecStart=/usr/local/bin/borg-backup.sh

# /etc/systemd/system/borg-backup.timer:
[Unit]
Description=Run Borg backup daily
[Timer]
OnCalendar=daily
Persistent=true
[Install]
WantedBy=timers.target

# Then: sudo systemctl enable borg-backup.timer

# Windows — Task Scheduler for rclone:
# schtasks /create /tn "Rclone Backup" /tr "rclone sync C:\Data D:\Backup"
#   /sc daily /st 02:00 /ru SYSTEM

# Cron for restic:
0 2 * * * /usr/local/bin/restic-backup.sh
# Runs at 2 AM every day

Offsite Storage Strategies

Offsite backups protect against physical disasters (fire, flood, theft):

  • Cloud backup — Easiest option. Backblaze B2, Wasabi, or S3 Glacier. Automatic, encrypted, always available. Cost: ~$5-10/TB/month.
  • Friend/family swap — Keep an external HDD at a trusted person's house. Swap monthly. Free but requires manual effort.
  • Safety deposit box — Best for archival backups (yearly). Not practical for frequent updates due to bank hours.
  • Office/second location — Keep backup at your workplace if you have a secure place. Rotate weekly.
  • Geographic diversity — Ensure offsite is at least 10 miles away. For cloud, choose a provider with data centers in different regions.
