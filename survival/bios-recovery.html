BIOS/UEFI Recovery

Recovering a system with corrupted BIOS/UEFI firmware - CMOS reset, Flashback, CH341A programmers, and debricking motherboards.

When BIOS Fails

A corrupted BIOS can brick your motherboard - no display, no POST, no response.

Causes: Failed update, wrong BIOS file, overclocking failure, physical damage, UEFI rootkit.

Symptoms: No display, no POST beeps, fans spin but nothing happens, boot loop.

Clearing CMOS

# Method 1: Clear CMOS jumper
# Find CLR_CMOS jumper, short pins 2-3 for 10s, return to 1-2

# Method 2: Remove battery
# Remove CR2032 battery, wait 5-10 min, reinsert

# Method 3: Clear CMOS button (high-end boards)

BIOS Flashback

BIOS Flashback allows updating BIOS without CPU/RAM/GPU.

# ASUS: rename BIOS file to CB****.CAP
# MSI: rename to MSI.ROM
# Gigabyte: rename to GIGABYTE.bin
# Format USB FAT32, insert in labeled port
# Press Flashback button, wait for LED to stop blinking

CH341A Programmer

The CH341A is a cheap (~$10) SPI flash programmer for direct BIOS chip read/write.

# Connect SOIC8 clip (pin 1 = red wire)
sudo apt install flashrom
sudo flashrom -p ch341a_spi -r backup.bin
sudo flashrom -p ch341a_spi -w new-bios.bin
sudo flashrom -p ch341a_spi -v new-bios.bin
Warning: Pin 1 mismatch can damage the chip. Double-check before powering on.

Recovering a Bricked Motherboard

Levels of debricking:

  1. Clear CMOS, minimal hardware, known-good PSU/RAM
  2. BIOS Flashback with correct file
  3. CH341A programmer - read, write, verify
  4. Hot flash (RISKY - identical donor board required)
  5. Buy pre-flashed chip from eBay (~$10-20)

BIOS vs UEFI Explained

Understanding the difference between BIOS and UEFI is essential for firmware recovery:

Feature Legacy BIOS UEFI
InterfaceText-based, keyboard navigationGUI with mouse support
Boot modeMBR (Master Boot Record)GPT (GUID Partition Table)
Boot speedSlow (must enumerate all hardware)Fast (parallel initialization)
Secure BootNot supportedPrevents unsigned bootloaders/kernels
Driver support16-bit, limited64-bit drivers, runtime services
Flash size1-4 MB8-32 MB
Chip typeUsually SOIC-8 LPC/FWHSOIC-8 or WSON-8 SPI

Most systems built after 2012 use UEFI. Legacy BIOS is found on older hardware and some embedded systems. Recovery methods differ significantly between the two.

When and Why to Backup Firmware

Backing up your current working BIOS/UEFI firmware before making changes is critical insurance:

  • Before updating firmware — If the update fails or the new version has bugs, restore the backup.
  • Before overclocking — If aggressive settings corrupt the UEFI configuration.
  • Before modifying UEFI — If you are editing UEFI variables, injecting drivers, or modding the firmware.
  • For dual BIOS boards — Copy the working ROM from the primary chip to use as a recovery source.

flashrom Usage Guide (Reading, Writing, Verifying)

flashrom is the universal tool for reading, writing, and verifying firmware chips. Here is the complete workflow:

# flashrom — Reading your BIOS chip:
# 1. Identify the chip:
sudo flashrom --programmer internal --identify
# Or with CH341A:
sudo flashrom -p ch341a_spi

# 2. Read (backup) the current firmware:
sudo flashrom -p internal -r bios-backup.bin

# 3. Verify the read:
sha256sum bios-backup.bin
# Read it twice and compare checksums

# 4. Write new firmware:
sudo flashrom -p internal -w new-bios.bin

# 5. Verify the write:
sudo flashrom -p internal -v new-bios.bin
# Compares the written data to the source file

# 6. Erase the chip completely:
sudo flashrom -p internal -E

# Common programmers:
# -p internal        : Built-in (run from OS on the target machine)
# -p ch341a_spi      : CH341A USB programmer
# -p ft2232_spi:type=2232H : FTDI FT2232H
# -p buspirate_spi   : Bus Pirate
# -p raspberry_hw    : Raspberry Pi GPIO
Warning: Writing the wrong firmware can permanently brick the motherboard. Always read and verify the original firmware before making changes. Keep the backup in multiple locations.

Raspberry Pi as a Hardware Programmer

A Raspberry Pi can act as a SPI flash programmer using its GPIO pins — cheaper than a CH341A and often more reliable:

# Raspberry Pi as SPI programmer:
# 1. Enable SPI: sudo raspi-config → Interface Options → SPI → Enable
# 2. Install flashrom:
sudo apt install flashrom
# 3. Connect SOIC clip to GPIO:
#    Pi Pin 19 (MOSI)  → Chip Pin 5 (MOSI/SI)
#    Pi Pin 21 (MISO)  → Chip Pin 2 (MISO/SO)
#    Pi Pin 23 (SCLK)  → Chip Pin 6 (SCLK/CLK)
#    Pi Pin 24 (CS/CE0) → Chip Pin 1 (CS/CE#)
#    Pi Pin 25 (GND)   → Chip Pin 4 (GND)
#    Pi Pin 17 (3.3V)  → Chip Pin 8 (VCC)
# 4. Verify connection:
sudo flashrom -p linux_spi:dev=/dev/spidev0.0,spispeed=1000
# 5. Read chip:
sudo flashrom -p linux_spi:dev=/dev/spidev0.0,spispeed=1000 -r backup.bin
# 6. Write chip:
sudo flashrom -p linux_spi:dev=/dev/spidev0.0,spispeed=1000 -w new-bios.bin

SOIC clip types: SOIC-8 (8-pin, most common desktop), SOIC-16 (16-pin, some laptops), WSON-8 (no-pin, requires pogo adapter). Always verify the chip package before buying clips.

CH341A Programmer Setup

The CH341A is the most cost-effective SPI programmer (~$10-15). Here is how to set it up safely:

# CH341A Setup Guide:
# Hardware:
# 1. CH341A programmer board (~$10 from Amazon/AliExpress)
# 2. SOIC8 test clip (red wire = pin 1)
# 3. 1.8V adapter (for modern low-voltage SPI chips)
# 4. Jumper wires (female-to-female)

# Software (Windows):
# 1. Install CH341A driver (from manufacturer or Zadig)
# 2. Download AsProgrammer or flashrom GUI
# 3. Connect clip to chip (double-check pin 1 alignment!)
# 4. Select chip model from list
# 5. Read → Save as .bin
# 6. Write → Select new firmware → Program

# Software (Linux):
sudo apt install flashrom
sudo flashrom -p ch341a_spi -r backup.bin  # Read
sudo flashrom -p ch341a_spi -w new.bin     # Write
sudo flashrom -p ch341a_spi -v new.bin     # Verify

# Voltage check:
# Most modern laptop BIOS chips are 1.8V
# Desktop BIOS chips are usually 3.3V
# Using 3.3V on a 1.8V chip will destroy it!
# Use voltage adapter or set CH341A jumper correctly
Critical: Always check the chip voltage before connecting. A 3.3V signal on a 1.8V chip will fry it instantly. Look up the chip datasheet or measure with a multimeter.

Bricked BIOS Recovery Methods

When the BIOS is corrupted and the system will not POST, try these methods in order:

  1. CMOS clear — Remove battery, short jumper, wait 10 minutes. Sometimes the corrupted settings are stored in CMOS RAM, not the flash chip.
  2. USB Flashback — Many modern motherboards (ASUS, MSI, Gigabyte) can recover from a specific USB port. Download the correct BIOS, rename per motherboard instructions, insert into the labeled port, press the Flashback button.
  3. Blind flash — If no Flashback button, try booting with a FAT32 USB containing the correct BIOS file. Some boards auto-detect and reflash from USB on boot (even without display).
  4. CH341A SPI programming — Clip onto the BIOS chip, read it, verify it is corrupt, flash the correct firmware. This works 90% of the time.
  5. Hot flash — Requires an identical working motherboard. Boot the working board, carefully swap the BIOS chips while powered on, flash the corrupt chip. EXTREMELY risky — one slip shorts pins and destroys both boards.
  6. Replace the chip — If the SPI chip is soldered, desolder it. Or buy a pre-flashed chip from eBay for $10-20. Search for your motherboard model listed with "BIOS chip."

UEFI Settings Backup via Shell

UEFI settings can be backed up and restored using the UEFI Shell (available on most UEFI systems):

# UEFI Shell — Backup and Restore Settings:
# 1. Access UEFI Shell:
#    - Some BIOS have a "Launch UEFI Shell" option
#    - Or boot from a UEFI Shell USB (download shell.efi)

# 2. Map drives:
Shell> map -r
# Shows available file systems (fs0:, fs1:, etc.)

# 3. List UEFI variables:
Shell> dmpstore -all
# Dumps all UEFI variables to screen

# 4. Save variables to file:
Shell> dmpstore -all -l fs0:\uefi-vars.txt
# Saves all variables to a text file on the USB

# 5. Restore specific variable:
Shell> setvar SecureBoot -guid xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx =0100

# 6. Key UEFI variables to back up:
# SecureBoot, SetupMode, db, dbx, KEK, PK
# These control Secure Boot and boot security

Secure Boot Management and TPM Issues

Secure Boot and TPM (Trusted Platform Module) can complicate firmware recovery:

  • Secure Boot — When enabled, only signed bootloaders can boot. To disable: BIOS → Boot → Secure Boot → Disable (or "Other OS"). To clear Secure Boot keys: BIOS → Secure Boot → Restore Factory Keys.
  • TPM issues after firmware update — The TPM is tied to the firmware. After a BIOS update, the TPM may lose trust with Windows. BitLocker will prompt for the recovery key. This is normal — enter the 48-digit key and Windows re-establishes trust.
  • Clearing TPM — Some BIOS have a "Clear TPM" option. This resets the TPM to factory state. All keys and secrets are lost (Windows Hello, BitLocker keys if not backed up). Only do this if you have recovery keys.
  • fTPM vs dTPM — fTPM (firmware TPM) is built into the CPU firmware. dTPM is a separate chip. fTPM is more common on modern AMD systems and can cause boot issues if firmware is corrupted.

BIOS Password Removal Methods

When a BIOS password prevents you from accessing or booting the system:

  • CMOS clear — Remove battery, short jumper, or use the clear CMOS button. This clears the BIOS password stored in CMOS RAM.
  • CMOS discharge — Remove battery and power cord, press the power button for 30 seconds to discharge capacitors, wait 10 minutes, reinstall battery.
  • Manufacturer backdoors — Some laptops have universal backdoor passwords based on the service tag or a specific keystroke sequence. Search for "BIOS master password [model]."
  • SPI chip reflash — If the password is stored in the BIOS flash (not CMOS), clear it by reading the flash with CH341A, flashing a known clean BIOS, or patching the password byte in a hex editor.
  • Soldered chip — On many modern ultrabooks, the BIOS chip is soldered. This requires careful desoldering or using a clip on WSON-8 packages.
  • Apple EFI password — Different from PC BIOS. Requires SMC/PRAM reset or professional EFI unlock (not DIY).

Dual BIOS Motherboard Recovery

Many Gigabyte and some ASUS motherboards have dual BIOS chips — a primary and a backup:

# Dual BIOS Recovery (Gigabyte):
# 1. If the system fails to POST 3 times, it automatically
#    switches to the backup BIOS
# 2. Manual switch: Power off → flip the DualBIOS switch
# 3. Boot from backup BIOS → flash the primary BIOS again
# 4. Flip switch back to primary → reboot

# Some boards have a physical switch on the PCB:
# Position 1: Primary BIOS
# Position 2: Backup BIOS
# Always boot from backup, flash primary, test

# If both BIOS chips are corrupt:
# 1. Identify both SPI chips on the board
# 2. Use CH341A to flash each one separately
# 3. Boot from backup chip after flashing primary

Best practice: Before flashing an experimental BIOS, manually flip to the backup BIOS, boot the system, and ensure the backup works. Then flip to primary and flash. If it fails, flip back to backup and recover.
